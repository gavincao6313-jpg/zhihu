- **[2026-06-22] Gemini API 调用计数不能硬编码 `conservative_calls = 1 + max_continuations`。** `batch_process_external.py` 原逻辑每个视频无论实际调几次 API 都按 3 次记账（`conservative_calls=1+max_continuations=3`），因为 `utils.call_gemini` 只返回文本不返回调用次数。修复：(1) `utils.call_gemini` 加 `_counter: list | None = None` 可选参数，调用结束前 `_counter.append(gemini_calls)` 回传真实次数；(2) batch 代码用 `_counter[0]` 替代硬编码 3。规则：任何包装 API 调用计数的代码必须返回实际次数，不得用 conservative estimate 作为账面值；pre-flight quota 预估可以保守，但 post-call 记账必须精确。

- **[2026-06-22] per-model 闸门 `expected_calls` 必须与 post-call 真实计数同步。** 修复 api_calls 后 post-call 记账已精确为 1，但 `gemini_expected_calls` 仍为 3（`1+GEMINI_BATCH_MAX_CONTINUATIONS`），导致每模型余额 2 时 16+3=19>18 被闸门拒绝，浪费 6 次配额。修复：`gemini_expected_calls` 同步为 1。规则：配额预检闸门的 `expected_calls` 必须与 post-call 实际计数一致，不得脱节。

- **[2026-06-22] 配额空窗期应本地预处理，配额恢复后直接推大模型。** 新增 `--preprocess-only` 参数：遍历 pending 视频，仅执行关键帧提取 + SenseVoice 转写，缓存到 `cache/`，不调用任何 API。配额恢复后正常运行，Phase 1 命中缓存秒过。效果：每视频从 7-12 分钟降至 1-2 分钟，日处理量从 18 暴增到 38（↑211%）。规则：批处理架构应分离本地预处理与 API 合成两个阶段；API 配额稀缺时不要闲置本地算力。
- **[2026-06-13] 批处理 `--source-dir` 必须与进度文件中 key 前缀一致，禁止用子目录。** `batch_process_external.py` 的 `discover_external_videos()` 以 `relative_to(source_dir)` 生成 stem 作为进度 key。若原处理用父目录（如 `E:\AI解决方案`，进度 key 为 `AI解决方案专家-七期-赠/001_xxx`），后续必须用同一父目录。用子目录（如 `F:\AI解决方案\AI解决方案专家-七期-赠`）会导致 stem 变为 `001_xxx`，与进度 key 完全不匹配，全部视频被视为新视频全量重处理。本次事故浪费 Qwen API 200+ 次调用，并因重处理中部分失败损坏 3 个已完好输出文件。规则：跑批处理前必须确认 `source_dir` 与进度 key 前缀的关系；如不确定，先 --dry-run 看 stem 列表是否匹配进度 key。
- **[2026-06-12] MAC 推送的脚本修复必须同步到 `zhihu_file/scripts/`。** `zhihu_file/` 是独立 git 仓库，头条脚本的实际运行副本在这里（`REPO_ROOT` 指向 `zhihu_file/`）。根目录 `scripts/` 的修复不会自动生效。本次 MAC 推送的 5 个修复（bug-167~170 + 临时文件清理）只修改了根 `scripts/`，必须手动应用到 `zhihu_file/scripts/`。规则：拉取头条相关修复后，先 `diff scripts/ zhihu_file/scripts/` 确认同步状态。
- **[2026-06-14] Gemini 无窗口化 + Qwen 审核阻断 = 大视频双重死锁。** `batch_process_external.py` 中 Gemini 路径全量发送 1000+ 帧超 1M token 限制（`INVALID_ARGUMENT`），Qwen 路径虽然分窗但个别窗口帧被内容审核拒绝（`data_inspection_failed`）→ Assembly 失败。两个模型各自能处理部分内容但无跨模型降级机制。修复方向：Gemini 加滑动窗口（`GEMINI_MAX_FRAMES` + `_synthesize_gemini_windowed`），Qwen 审核失败窗口用 Gemini 补位。详见 `docs/BUG_batch_gemini_no_windowing_qwen_audit_block.md`。规则：批处理启动前必须检查是否有大视频（>1000帧）被路由到 Gemini，如有则必须先改 provider 为 qwen 或等待 Gemini 窗口化实现。

- **[2026-06-04] 混合合成架构方向：Qwen 滑动窗口 + Gemini 最终一次调用。** 当前问题：Qwen 滑窗保真度好但输出质量不如 Gemini；Gemini 一次全量合成输入 ~280k tokens 超 TPM、续写最多 21 次耗尽配额。新架构：(1) Qwen 滑动窗口对全量帧/转写多轮提取笔记（保真、多次 Qwen 调用、低成本）；(2) Gemini 接收 Qwen 笔记 + 精选关键帧做最终一次高质量合成（Gemini RPD 固定消耗 1 次）。预期收益：Gemini 从 2-21 次降为恒定 1 次，输入信噪比提升，TPM 超限风险降低。重构完成后需重新对比 API 消耗与输出质量，参考本次基准数据（Key Learnings: API 配额消耗基准 2026-06-04）。
- **[2026-06-04] Gemini File API 直传视频已验证不可用。** `experiment/inline-and-uri-upload` 分支测试了 base64 inline、`file_uri` 两种上传模式（`UPLOAD_MODE` 环境变量切换）。结论：Gemini File API 直传本地视频文件在当前项目架构下不可行——`file_uri` 模式要求文件在 Google 自家云存储（Google Drive / Google Cloud Storage）中，不能直接引用本地路径或自建 CDN URL；base64 inline 受 token 上限和 Free-tier TPM 限制，对 2h+ 直播视频完全不可用。**正确方案仍是 ASR + 关键帧分片，通过文本+图片 parts 提交 Gemini**，不走整视频直传路径。分支 `experiment/inline-and-uri-upload` 已于 2026-06-04 删除。
- **[2026-06-02] 产品方向转型：从后端 CLI 工具演进为完整软件产品。** 用户明确表示当前始终是一个后端工具，不是完整的软件功能。下一阶段核心目标是设计并实现前端可视化控制台，将整个直播转写流程可视化：启动过程、运行记录、中间产物（chunk/keyframe/QC）、逐字稿、最终输出 MD 文件。前端设计工作计划于 2026-06-03 启动。
