# AI 审查工作流

适用于 L3 及核心 L2 变更，由 CLAUDE.md 变更分级控制章节引用。

## 第一步 —— 生成 Diff

确认工作区干净后，生成基于正确基线的 diff：

```bash
git status                                                # 确认工作区干净
git diff --merge-base main HEAD > ai_review_diff.patch    # 生成分支相对于合并基点的差异
```

如果未提交的暂存改动也需纳入审查：

```bash
git diff --staged >> ai_review_diff.patch
```

不要使用裸 `git diff main`——若本地 `main` 落后于 `origin/main`，基线可能不准确。

## 第二步 —— 组装审查包

不要只发裸 diff。带上必要上下文，避免外部 AI 产生大量低价值误报：

```
变更目标：[一句话描述这次改动做了什么]
风险级别：[核心 L2 / L3]
相关约束：
- Gemini Free Tier：gemini-2.5-pro 5 RPM/100 RPD；gemini-2.5-flash 10 RPM/250 RPD
- Windows 只负责运行脚本，Mac 负责代码改动
- [补充本次变更涉及的其他约束]
已执行的检查：[列出实际跑了哪些]
未验证项：[哪些检查没跑，原因是什么]

[粘贴 diff]
```

## 第三步 —— Auditor Prompt

打开一个无项目上下文的全新 AI 会话，粘贴以下 prompt，后面跟上第二步的审查包：

---

你是一位资深架构师和安全专家，正在审查一个 Python 视频与直播流处理项目的代码变更。请对以下改动进行严格、精简的审计，重点关注：

1. **范围蔓延**：执行者是否在目标之外，顺手重写了周围原本稳定的代码？是否有过度设计？
2. **边界漏洞**：是否存在未处理的 null/空值、异步竞态、未捕获异常、资源泄漏——尤其在流切片、长任务运行、重试循环、checkpoint/manifest 逻辑中？
3. **成本与性能**：是否存在重复的 Gemini API 调用、无上限的重试、O(N²) 循环、硬编码的凭据或 API Key？
4. **代码污染**：是否残留调试用的 print、未使用的变量、宽泛吞异常的 except、或死代码？

每个发现请给出：文件和大致行号、风险等级（低/中/高）、修复建议。对于明显符合变更目标和已知约束的代码，跳过不报。

[粘贴审查包]

---

## 第四步 —— 对质与闭环

人类（法官）过滤误报，将有效发现派发给 Driver：

> "外部审计发现了以下问题：[发现内容]。请精准修复，不要引入新的改动范围。"

Driver 修复后重新跑最低验证，人类确认后合并。

## 外部 AI 审查不能替代的事项

- 改动任何 symbol 前必须跑 `gitnexus_impact` impact analysis
- 提交前必须跑 `gitnexus_detect_changes()`
- 本机实际验证运行
