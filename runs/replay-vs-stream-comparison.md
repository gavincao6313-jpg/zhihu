## 二、时间戳精度对比

### 回放转写 (本次)
- **方式**: 整段转写后按字数比例估算每句话的时间戳
- **精度**: 近似值，误差随累积偏移增大
- **示例**: `[00:00:03] 这样有声音吗？` (估算值)
- **问题**: 
  - 无法反映语速变化（快/慢/停顿导致偏移）
  - 长文本累积误差可达数十秒
  - 无法精确定位到音频中的具体时间点

### 直播流转写
- **方式**: 按60秒固定分片，每个分片独立转写
- **精度**: 精确到分片边界 (±60秒)
- **示例**: `[00:01:00 - 00:02:00] 确实不太懂...`
- **优势**: 
  - 分片边界精确对应音频位置
  - 适合需要快速定位回看原视频的场景
  - 每个分片独立，不存在累积误差

**结论**: 直播流转写的时间戳精度**显著优于**回放转写。

## 三、内容完整性对比

### 回放转写
| 优势 | 劣势 |
|---|---|
| 覆盖完整 2h51m 内容 | 整段输出缺少结构层次 |
| 包含开场音频调试等"无效"内容 | 无法跳过空白/静音区域 |
| 内容无遗漏 | 所有内容揉在一起 |

### 直播流转写
| 优势 | 劣势 |
|---|---|
| 每分钟一个独立块，结构清晰 | 只捕获了 66/171 分钟 |
| 可追踪处理状态和进度 | 存在分片边界处截断的风险 |
| 实时处理，直播结束后即有输出 | 依赖网络稳定性 |

**结论**: 回放转写**内容更完整**（全量），但**结构性较差**（无分片）；直播流转写**结构清晰**但**覆盖不全**。

## 四、转写质量（准确率）

两者使用相同的 SenseVoiceSmall 模型，核心转写准确率**基本一致**。

| 观察点 | 说明 |
|---|---|
| **中文识别** | 两者均表现良好，无明显差异 |
| **情感标签** | 两者均保留了 SenseVoice 的情感标记 (😊, 😡 等) |
| **专有名词** | 两者在相同位置出现类似错误 (如 "prompput" → 应为 "prompt") |
| **重复/口吃** | 两者均保留了说话人的口语特征 (重复、修正) |
| **背景噪音** | 回放视频可能存在二次编码损失，略逊于直播流原始音频 |

**关键发现**: 回放转写因 `merge_vad=True` 将整段音频合并，导致失去了句子级的时间戳信息。SenseVoice 的 `sentence_info` 字段未被正确利用。

## 六、方法论对比总结

| 特性 | 回放离线转写 | 实时流转写 |
|---|---|---|
| **适用场景** | 事后完整回放分析 | 直播实时监控/转写 |
| **输入** | 完整MP4文件 | FLV直播流 (HTTP) |
| **分片策略** | 全量一次处理 | 60秒增量分片 |
| **延迟** | 无实时性要求 | 60秒内产出 |
| **失败恢复** | 需整体重跑 | 单分片重试 |
| **资源占用** | 短时高负载 | 长时间低负载 |
| **输出时效** | T+10min | T+60s |

---

## 七、改进建议

### 对回放转写流程
1. **分段转写**: 将长视频按60秒分片后再转写，获得精确时间戳
2. **VAD优化**: 使用 `merge_vad=False` 保留句子级边界
3. **sentence_info利用**: 直接从 SenseVoice 返回结果中提取 `sentence_info` 获得原生时间戳
4. **内容结构化**: 利用幻灯片切换事件生成有意义的章节标题

### 对直播流转写
1. **完整性**: 增加流中断检测，确保覆盖整个直播
2. **后处理**: 将逐分钟块合并为更长的语义段落 (5-10分钟)
3. **内容摘要**: 利用关键帧事件自动生成每个章节的摘要
