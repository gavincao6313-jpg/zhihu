# Skill模式的解析，和agent的关系

## 1. 视频元数据
- **推测主题：** 本视频深入解析了AI领域中“Skill模式”的概念、实现机制、与传统Agent和Workflow模式的异同，并探讨了Skill模式对AI产品设计和开发带来的五个核心变革。
- **核心关键词：** Skill模式, Agent, Workflow, Prompt Engineering, LLM, AI产品设计, Coze, YAML头, SOP, Multi-Agent, 任务编排, 技能市场, 产品思维。
- **适用受众/场景：** 适用于AI产品经理、AI开发者、技术架构师以及任何对AI Agent和Skill模式感兴趣的人员，旨在解决AI产品开发中灵活性、可维护性、成本控制和功能扩展等问题。

## 2. 核心知识字典（Glossary）

-   **Skill模式 (Skill Mode)**: 一种将特定任务或能力封装成标准化文件结构（通常是一个包含Markdown描述和YAML头的文件目录）的方法。Agent通过读取这些文件的YAML头来识别和按需调用相应的技能，从而执行复杂任务。它类似于一个操作手册或工具箱，赋予Agent特定能力。
-   **Agent (智能体)**: 在AI领域中，Agent可以被理解为一个“人”或“智能体”，它能够理解用户意图，拆分任务，并根据需要调用不同的工具或技能（如Skill）来执行任务并返回结果。
-   **Workflow (工作流)**: 一套流程化的系统，包含多个步骤，每个步骤可以调用不同的大模型API、工具或Agent，用于回答问题或完成某项任务。其核心是编排逻辑，旨在复刻人类脑力劳动的流水线，强调确定性和多Agent间的消息传递。
-   **YAML头 (YAML Header)**: Skill文件（通常是`skill.md`）开头的一段YAML格式的元数据，包含Skill的名称（name）和描述（description）。大模型在加载Agent时会预加载所有Skill的YAML头到其上下文，以便在需要时识别并调用相应的Skill。
-   **Prompt Engineering (提示词工程)**: 编写和优化提示词（Prompt）以引导大模型生成期望输出的技术。在Skill模式中，Prompt仍然是基础，但Skill通过结构化封装和流程定义，减少了对单一长Prompt的依赖，提高了输出的稳定性和可维护性。

### [00:05:47 - 00:42:54] Skill模式是什么，有什么场景

-   **核心论点：** Skill模式是一种将特定任务或能力封装成标准化文件结构的方法，它通过轻量级的单Agent模式实现多Agent的优势，解决了传统Prompt和Workflow在复杂场景下灵活性、维护性和成本的问题。
-   **详细展开：**
    *   **“同事.skill”现象 (00:06:43)**：讲师从网络上流行的“同事.skill”梗引入，讨论是否能将同事或前任“整流”成Skill。他认为，前任可以被“整流”成Skill（因为核心是语气和关键时间点），但同事（尤其是从事复杂、难以描述工作内容的同事）很难被单一Skill完全替代，可能需要多个Skill。这引申出在职场中保持竞争力的关键：如果你的工作能轻易被一个Skill替代，则竞争力不足。
    *   **Skill的火爆与应用 (00:09:45)**：Skill模式在近期非常火爆，Coze（扣子）平台已上线技能商店，提供大量Skill，涵盖工具、开发、数据、AI、商业、运维、测试、安全、内容媒体、文档研究、生活方式等领域。
    *   **Coze官方新年绘本案例 (00:10:24)**：讲师展示了Coze官方的一个新年绘本Skill。用户输入主角信息（如“豆豆”，一个可爱的两岁小女孩），Skill会：
        1.  构建完整故事脚本。
        2.  拆分分镜。
        3.  为每个分镜生成图像提示词（Image Prompt）。
        4.  编写对应页面的文字。
        最终生成一个可自动播放、带音乐、翻页功能、一键打印的交互式HTML电子书。这表明Skill内部包含一个明确的工作流程。
    *   **Skill与Workflow的对比 (00:13:15)**：
        *   **概念交集**：讲师强调，AI领域的新框架和模式（如Skill、Workflow、Prompt Engineering、Agent、HearthMos等）并非正交关系，而是彼此有交集，甚至可以相互嵌套（例如，Skill可以内嵌Workflow、MCP、Function Call、RAG）。
        *   **Prompt的问题 (00:16:57)**：Prompt本质上是一次性上下文，多个Prompt组合容易互相干扰，难以解决“以后遇到类似问题如何回答”和“本轮遇到其他问题怎么办”的长期性和稳定性问题。
        *   **Workflow解决的问题 (00:18:02)**：Workflow核心是确定性，旨在复刻人类脑力劳动流水线，通过多个Agent（每个Agent负责特定任务，如选题、文案、分镜等）协作完成复杂任务。它强调编排逻辑和预设流程，但确定性与灵活性互斥。
        *   **Workflow的局限性 (00:22:47)**：在开发个人助理等需要多种能力（订机票、订酒店、做报告、数据分析、心理陪伴等）的复杂场景时，传统Workflow方案会变得非常复杂。需要前置意图识别、多个子工作流、消息重排、Prompt腐化等，导致维护困难、Token消耗大，结构庞大如树状图。
    *   **Skill解决的问题 (00:26:05)**：
        *   **核心**：让大模型记住如何做一件事，将需求转化为标准化的文件结构。
        *   **三个核心**：
            1.  用一个文件夹封装一个“技能”（`skill-name/`）。
            2.  用Markdown描述这个“技能”的调用方式（`skill.md`）。
            3.  用YAML头+索引，让大模型知道什么时候调用。
        *   **Skill的实现机制 (00:30:07)**：Agent在初始化时，会将所有Skill文件夹中的`skill.md`文件的YAML头（包含name和description）预加载到其上下文窗口。当大模型在对话中识别到用户意图与某个YAML头匹配时（例如，用户说“数据分析”，命中`name: data-analysis`），它会按需调用该Skill的完整Markdown正文，并根据正文描述完成后续任务。
        *   **Skill的优势 (00:32:20)**：
            *   **灵活性**：可不断增加Skill，使Agent能力更强，无需修改核心Prompt。
            *   **专业性**：单一Skill可以做得非常强大，可以封装Workflow、MCP等。
            *   **成本**：初始只读取YAML头（例如50个Skill仅需5000 Token），不读全文，Token消耗低。
        *   **Skill与Agent的关系 (00:42:21)**：Agent是“人”（智能体），Skill是“操作手册”。Agent根据任务需求查阅并执行相应的Skill手册。例如，飞行员（Agent）在紧急情况（如空难）下查阅操作手册（Skill），以牺牲灵活性换取确定性，避免幻觉。机长作为MCP（Master Control Program）负责控制外部操作杆，同时指令副机长（副Agent）查阅手册（Skill）。
        *   **Agent开发阶段的演进 (00:52:53)**：
            1.  **单体Agent**：一个超长的System Prompt + RAG + MCP。问题是Prompt越写越长，维护困难，难以复用，Token消耗巨大。
            2.  **Multi-Agent架构**：一个总控Agent + 多个专职Agent。主Agent理解用户目标，拆分任务，分配给子Agent。子Agent负责规划、执行、评审、文案、绘制等。问题在于路由和编排复杂，需要大量规则，信息在Agent间传递时容易丢失或变形，每个Agent都有独立上下文，导致上下文切换成本高。
            3.  **Skill模式**：不更换Agent，只给当前Agent更换“角色+手册”。Skill是一个可插拔的“角色配置+操作手册+工具调用套路”，在需要时按需加载到Agent的上下文。它用轻量级单Agent模式实现了多Agent的好处，减少了进程和消息传递成本。
    *   **Skill的缺点 (00:55:04)**：Workflow在追求高确定性和每个环节专业性（如心理学分析、导演分镜）的固定化流程中，仍具有优势。Skill和Workflow是平衡博弈和选择，取决于对灵活性和确定性的需求。
    *   **适合的场景 (00:59:08)**：
        *   **个人助理**：日程管理、邮件发送、机票订单、报销对账、家庭订单、医疗挂号等。
        *   **SaaS**：线索收集-入库-外呼、商机报价-审批-下单、采购-比价-收货、费用报销、法务、财务等。

### [01:00:27 - 01:59:40] Skill的制作和使用方法

-   **核心论点：** Skill的创建可以通过自然语言自动生成，也可以手动编写，关键在于清晰定义其用途、触发方式、输出要求和所需资源。实际操作中，通常是先由大模型生成初稿再进行修改。
-   **详细展开：**
    *   **Coze技能商店 (01:00:33)**：Coze上已有不少付费Skill，如“专利小灯塔一键出稿”、“小红书笔记一键生成”等。这些商店中的Skill文件通常不开源，以保护商业价值。
    *   **通过自然语言创建Skill (01:02:21)**：
        *   **案例**：创建一个“副业教练”Skill。用户输入技能特点、时间投入、启动资金、城市、变现方式、收入目标、当前工作等信息。Skill需要提供三个可选方向（包含技能要求、时间投入、预期收入、难点要点），用户确认后设计完整工作计划和每日督导计划。
        *   **Coze自动生成过程 (01:04:26)**：Coze会根据自然语言描述自动创建Skill文件（如`Side Hustle Coach/skill.md`），并生成配套的参考文件（如`side-directions.md`副业方向库、`planning-template.md`计划模板、`coaching-framework.md`教练计划框架）。
        *   **`skill.md`文件结构 (01:05:22)**：
            *   **YAML头**：`name: Side Hustle Coach`，`description: 帮助用户规划副业变现技能，根据技能和时间资源提供三个方向建议，设计完整工作计划，并每日督导。`
            *   **正文**：包含任务目标、触发条件、前置准备、Skill要求、操作步骤（信息收集、分析匹配、用户确认、制定计划、每日督导）、资源索引、注意事项、使用示例等。
        *   **使用体验 (01:20:50)**：用户输入“我想做个副业”，Agent命中Skill的YAML头，然后根据Skill的“信息收集”步骤，询问用户详细信息。用户提供信息后，Agent根据“分析匹配”步骤和“副业方向参考库”给出三个副业方向建议（如产品经理专栏作者、知识星球运营、小红书产品博主），并提供详细计划。
        *   **关键点**：自然语言描述需求越清晰，生成的Skill质量越高。Skill文件是物理文件，存储在云端目录，可下载修改。
    *   **手动创建Skill (01:32:31)**：
        *   **支持环境**：Cloud Code, Code Chat, Gemini CLI, Open Code, AMP, Code Buddy, Antigraph, VS Code, Coze CLI, Q-Code, Trail, Windward, Kilo, Factory, Goose, Coze (特殊方式), Manus (特殊方式), Cloud Desk (特殊方式)等IDE或CLI工具。
        *   **核心要素**：
            1.  **用途 (Purpose)**：Skill要解决什么问题，例如“获取英文博客URL并将其转换为Markdown文件，同时将英文Markdown文件翻译为中文”。
            2.  **触发方式 (Trigger)**：Agent在什么条件下调用此Skill，例如“当用户想要保存并翻译博客文章时使用”。这主要通过YAML头中的`name`和`description`来定义。
            3.  **输出要求 (Output Requirements)**：明确期望的输出格式、命名规范、保存位置等，避免AI发散。
            4.  **所需资源 (Required Resources)**：Skill执行所需的脚本、参考文档、API接口（如MCP、Function Call）等。
        *   **案例：英文博客翻译Skill (01:39:30)**：
            *   **Prompt描述**：详细说明Skill功能（获取英文博客URL，转Markdown，翻译成中文，保存本地），核心流程（Python脚本抓取、Prompt翻译、润色），输出文件要求（英文/中文Markdown，命名格式，目录结构），触发条件。
            *   **自动创建**：将上述Prompt输入到支持Skill创建的IDE/CLI（如Closer的`crackskill`技能），它会自动创建Skill目录结构，生成`skill.md`、`referenced.md`、Python脚本（`fresh_blog_to_md.py`用于抓取博客内容）等文件。
            *   **使用效果 (01:43:13)**：用户输入“保存并翻译博客文章”，提供URL。Agent会按Skill步骤执行：检查Python环境（若无则自动安装），抓取英文博客内容，转换为Markdown，进行翻译和润色，生成中文Markdown，并按预设格式保存文件。
    *   **“整流”员工工作为Skill的五个步骤 (01:46:55)**：
        1.  **抽象SOP (Standard Operating Procedure)**：将人脑的动作过程抽象为具体步骤。例如，竞品调研工作：阅读文章 -> 过滤废话 -> 提取关键数据 -> 与公司对比 -> 得出结论。
        2.  **员工属性参数化 (Parameterization)**：定义Skill的API接口，区分系统固化属性（如角色性格）和每次运行动态传入的参数（如分析维度）。例如，输入URL（必填）、关注维度（可选，有默认值如“核心功能”、“定价”、“目标人群”）。
        3.  **系统提示词的结构化封装 (Structured System Prompt)**：定义Agent的角色（Role）、目标（Objective）、约束（Constraints）和输出格式（Output Format）。其中，`Constraints`（绝对不能干啥）和`Output Format`（长什么样）是关键，用于限制大模型的发散性，确保输出质量和精度。例如，报告严禁使用“总是言之”、“首先其次”等废话，数据未披露必须输出“未披露”，严禁推测，语气采用咨询公司风格。
        4.  **外部知识库挂载 (External Knowledge Base)**：将公司内部知识（如公司产品路线图、市场数据）存储在RAG中，让Skill在分析过程中调用，使输出更贴近公司实际情况。
        5.  **记忆机制设计 (Memory Mechanism)**：设计外部键值数据库或利用框架自带的Memory模块，存储用户偏好。例如，用户修改了关注点，下次默认值会自动更新。
        *   **完整Skill结构 (01:58:15)**：包含元数据（name, description, version）、认知核心（role_setting, constraints）、输入输出约束（inputs, output_format）、动作与边界（tools, workflow_sop）、记忆配置（memory）。
        *   **使用效果 (01:59:01)**：Agent根据竞品发布会URL，按照Skill的规约输出结构化报告，包含表情符号、关键变化、敏感点、员工建议等，并附带数据来源。

### [01:59:41 - 02:31:48] Skill在AI产品设计上的影响和变化

-   **核心论点：** Skill模式将深刻影响AI产品设计的五个方面：任务编排思路、业务逻辑解耦、交互设计组件化、技能市场平台化以及任务成功率的价值评估，最终可能促成LLM智能中台的出现。
-   **详细展开：**
    *   **任务编排思路的变化 (02:02:27)**：
        *   从通用对话设计转向任务编排。Agent不再是“什么都能聊的通才”，而是拥有各种工具箱的“协调者”。
        *   设计中心从“让AI更像人，完成更多类型工作”转向“定义原子技能边界”，用Skill封装最小的、确定的任务。
        *   产品经理的核心价值在于理解业务流程，并将其拆解为一个个可调用的Skill，以及规划Skill的架构和边界。
    *   **业务逻辑层、推理层和执行层的解耦 (02:08:55)**：
        *   **灵活性更高**：LLM底座可以随时更换，业务代码保持不变，Skill接口标准一致。
        *   **确定性更高**：Skill模式将不确定的大模型输出约束在确定的逻辑里，Agent整体稳定性更好。
    *   **交互设计的组件化和去对话化 (02:10:24)**：
        *   Skill可以封装一套UI组件，实现卡片式交互（例如，灵光的卡片式交互）。
        *   当用户触发特定Skill（如订票），Agent不再返回纯文本，而是构建一套界面结果（如行程卡片）。
    *   **技能市场的平台化 (02:11:24)**：
        *   Skill模式使产品扩展性极佳，Agent可快速具备多种功能，实现乐高式构建。
        *   类似于GPTs或MCP，插入一个Skill就具备一块能力。
        *   既保持用户端的灵活性，又可构建生态系统。Workflow难以实现真正的平台化。
        *   市场上的公版Skill可以作为基础，但核心竞争力在于定制化和对用户需求的准确定义。
    *   **任务成功率的价值 (02:14:56)**：
        *   评价体系从“语义准确性”转向“任务成功率”。
        *   对话模式的核心指标是语义关联性和幻觉率，调试重点是提示词工程。
        *   Skill模式的核心指标是调用准确度和任务达成率，调试重点是接口描述优化（IO边界），错误处理需要异常捕获和回退机制。
        *   AI产品从“顾问”角色转变为“干活的”。
    *   **Agent设计的变化 (02:16:10)**：Skill将成为LLM智能中台化的可能性，组织结构可能出现智能中台，负责设计可复用的Skill。组件化设计为模块拆分提供了帮助。
-   **产品思维的提升 (02:23:25)**：讲师分享了自己提升产品思维的方法：像“魔怔人”一样，对看到的任何产品（无论是实体鼠标、手柄还是AI产品）都进行深度拆解和思考。包括：用户价值、产品设计流程、功能细节、营销卖点、成本结构、竞品分析、版本迭代原因等。通过不断代入产品经理角色进行思考，可以快速提升产品能力。

## 4. 遗留问题与下一步行动（如有）

-   **遗留问题：**
    *   如何更有效地判断一个业务流程应该拆分为Skill还是Workflow，或者如何平衡两者的使用？
    *   在Skill模式下，如何处理大模型在执行过程中可能出现的“幻觉”或错误，除了测试和异常捕获/回退机制外，是否有更主动的预防措施？
    *   如何衡量和提升产品经理在AI时代“准确定义需求”的能力，并将其体现在简历和面试中？
    *   在Skill市场中，如何评估和选择高质量的开源或付费Skill，以及如何避免过度依赖公版Skill导致产品缺乏核心竞争力？
    *   Skill的架构设计（如文件夹结构、Skill边界划分）的具体最佳实践和方法论。
-   **下一步行动：**
    *   建议读者尝试使用Coze等平台，通过自然语言创建和部署自己的Skill，亲身体验Skill的制作和使用流程。
    *   深入学习提示词工程（Prompt Engineering），因为它是所有AI框架和模式的基础。
    *   培养产品思维，对日常接触到的产品进行拆解分析，思考其用户价值、设计逻辑和迭代策略。
    *   研究Skill的架构设计，理解如何将复杂的业务流程合理地拆解为可调用的Skill模块。
    *   关注Skill市场（如skillsmp.com），了解行业内Skill的发展趋势和应用案例。
