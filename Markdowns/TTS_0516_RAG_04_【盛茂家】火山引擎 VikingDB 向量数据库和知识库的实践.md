## 2. 核心知识字典（Glossary）

- **向量数据库 (Vector Database):** 一种专门用于存储、管理和检索向量（Embedding）数据的数据库。它通过计算向量之间的相似度来支持语义搜索、推荐等AI应用。
- **Embedding (向量):** 将文本、图像、音频、视频等非结构化数据转换成高维数值向量，这些向量能够捕捉数据的语义信息，使得机器可以理解和处理这些数据。
- **ANN (Approximate Nearest Neighbor) 近似最近邻搜索:** 一种在海量高维向量数据中快速查找与查询向量最相似（或最近）的K个向量的算法。与精确最近邻（KNN）相比，ANN牺牲了一定的精度以换取显著的查询速度。
- **RAG (Retrieval-Augmented Generation) 检索增强生成:** 一种结合检索系统和生成式大语言模型（LLM）的技术。它首先从知识库中检索相关信息，然后将这些信息作为上下文输入给LLM，以生成更准确、更少幻觉的回答。
- **VikinDB:** 火山引擎自研的向量数据库产品，强调高性能、大规模检索、低成本和云原生特性，并提供知识库服务。

### [00:02:26 - 00:07:20] 向量数据库发展历史与行业概况

- **核心论点：** 向量数据库的发展是随着AI算法的进步而逐步演进的，从早期基础技术开源到大模型时代的爆发式增长，并伴随着行业内的技术争论。
- **详细展开：**
    - **1.1 向量数据库行业发展**
        - **2017-2018年：** 向量数据库的基础技术逐步出现。标志性事件包括Facebook（Meta）开源的Faiss（17年）和Google的Bert算法（最早使用Attention架构）。这证明了Embedding在搜索领域具有效果优势。
        - **应用：** 主流搜索厂商如Google、Meta、头条搜索等开始大规模落地基于向量检索的语义检索能力，作为关键词/全文检索（如ES）的补充。
        - **学术界：** 出现NN Benchmark等开源标准来评估向量检索性能。
        - **算法：** 早期以向量聚类（如IVF）为主，18年左右HSW算法成熟并展现出巨大优势。
        - **2019年起：** 各大公司（字节、阿里、百度、Google、微软）开始建设新一代向量检索引擎，如字节的Viking、阿里的Proxima、百度的Puck等。第一波向量数据库创业公司如Zilliz（Milvus）、Pinecone也在此阶段成立。
        - **2022年之前：** 向量数据库主要应用于搜索、广告、推荐等场景，算法技术持续发展使其在多模态检索、跨模态检索中得到应用。OpenAI的Clip模型是跨模态检索的典型例子。向量数据库作为非结构化数据处理的协处理器开始获得关注，因为非结构化数据占计算机领域数据总量的80%。
        - **2022年GPT3.5/ChatGPT出现后：** 引爆了自然语言处理领域，并推动RAG模式的普及。随着多模态大模型的发展，非结构化数据加工和处理迎来变革，对非结构化数据检索需求增加，向量数据库受到更广泛关注。传统数据库厂商（如ElasticSearch、PostgreSQL）也开始切入向量检索领域。
    - **行业公司列表：**
        - **开源产品：** Chroma（轻量级，嵌入式数据库，适用于RAG Demo、智能体本地临时记忆）、Qdrant（能力完备，多语言支持，适用于轻量级海外应用开发）、Milvus（能力完备，多语言支持，适用于轻量级海外应用开发，国内商业化）、LanceDB（列存优先，侧重多模态大数据生态，适用于多模态分析应用）、ElasticSearch（大容量，稳定性相对成熟，适用于对时效性要求不高的全文搜索任务，由云厂商提供）。

### [00:07:20 - 00:19:50] 向量数据库行业发展中的两大争论

- **核心论点：** 向量数据库行业存在两大主要争论：是否需要独立的向量数据库产品，以及向量检索接口应采用RESTful API/DSL还是传统SQL。
- **详细展开：**
    - **争论1：向量检索能力是否应独立封装为产品？**
        - **正方观点：** 向量检索具有特殊性，需要针对性优化，涉及非结构化数据大趋势和算法技术变革。传统数据库设计模式限制了其在更大挑战下的适应能力。
        - **反方观点：** 向量只是一种数据类型，可直接嵌入现有数据库作为Feature，降低数据迁移和用户学习成本。
    - **争论2：向量检索接口应采用RESTful API/DSL还是传统SQL？**
        - **正方观点：** 传统SQL不适合非结构化数据检索，应采用更适应非结构化数据检索的RESTful API或领域特定语言（DSL），提供更简单、可信赖的产品。
        - **反方观点：** 传统数据库扩展应以SQL为首选，通过UDF或单独算子支持向量检索，降低学习成本。
    - **讲师观点：** 作为使用者，向量数据库应专注于AI应用，提供简单、可信赖的接口（如RESTful API），而非复杂的SQL封装，以满足Agent或知识库等场景的快速开发需求。

### [00:19:50 - 00:29:55] 向量数据库原理入门：理解向量与向量检索

- **核心论点：** 向量是机器语言，能够表达多模态非结构化信息，其维度增长带来表达能力增强。向量检索通过距离计算找到最相似的TopK结果。
- **详细展开：**
    - **1.2.1 理解向量与向量检索**
        - **向量的出现：** 随着NLP算法发展，从TD-IDF稀疏向量到Word2Vec稠密向量，再到Bert Embedding和GPT Embedding，向量的表达能力不断增强。
        - **向量维度变化趋势：** 早期64维，逐步扩展到256维，目前主流为512-2048维，甚至有4096维。但并非维度越高越好，例如4096维相比2048维在模型架构不变时，可能无法带来额外收益。
        - **向量表达能力：** 维度增长带来表达能力增强。例如，2维表达地点，3维表达空间，4维（加时间）表达时空中的一个点。高维向量甚至可以恢复原始文本（在特定条件下）。
        - **多模态数据：** 文本、图片、PDF、视频等不同体裁的数据都可以转换为向量，表明向量是机器之间通用的语言。
        - **向量相似度指标：** 评估两个向量相似度常用欧氏距离、点积相似度、余弦相似度、Jaccard距离。其中，点积相似度在向量归一化后等价于余弦相似度。
        - **向量检索任务：** 从海量候选向量中，找到与查询向量最相似（距离最近）的TopK个候选。通常返回ID和相似度分数。
        - **关键信息：** 向量是机器语言，可表达任意非结构化信息。LLM中提取的Embedding表达能力惊人。

### [00:29:55 - 00:34:54] 手搓向量数据库基础架构

- **核心论点：** 即使是简单的向量数据库，也需要考虑索引、数据存储、查询解析和高级特性（如过滤）等模块。
- **详细展开：**
    - **2. 手搓向量数据库基础架构**
        - **模块划分：** 最简版的向量数据库可分为：Query Parse (查询解析)、ANN/KNN Engine (向量检索引擎)、Metadata Filtering (元数据过滤)、Index Mgr (索引管理)、RPC Server Framework (RPC服务框架)、DataStore (数据存储)。
        - **ANN/KNN Engine：** 实现向量检索，小规模数据可直接使用Numpy进行矩阵乘法和排序实现KNN。对于ANN，可使用Faiss或HNSW等开源库。
        - **Metadata Filtering：** 基于JSON格式的元数据，可使用Pandas或Polars等DataFrame库进行内存过滤，然后与ANN结果求交。
        - **Query Parse：** 通过RESTful API或自定义DSL支持查询。RESTful API更轻量易用。
        - **Index Mgr：** 负责索引的持久化、多索引管理、新数据插入后的索引重建等。
        - **DataStore：** 支持本地文件存储或对象存储（如OSS），实现“云原生”能力，服务无状态化，便于故障恢复。
    - **小结：** 对于数据量小于100万条，QPS低于100的场景，手搓一个向量数据库是可行的。但随着业务规模增长，复杂性会显著增加。

### [00:34:54 - 00:51:50] 度量向量数据库的能力

- **核心论点：** 评估向量数据库能力需关注延时、QPS、查询精度、写入性能、容量、数据安全等多个维度，尤其在大规模、复杂场景下，商业级服务相比自建方案具有显著优势。
- **详细展开：**
    - **3. 如何度量向量数据库的能力**
        - **Benchmark：** 现有ANN Benchmark主要针对纯向量检索性能，难以全面评估复杂数据分布、过滤条件和大规模场景下的表现。建议在遇到性能挑战时，选择商业服务或自行优化。
        - **关键指标：**
            - **延时：** 数据规模增大时，能否持续保持O(logN)的时间复杂度（如VikinDB可做到1亿数据5毫秒），避免退化到线性复杂度。
            - **最大QPS：** 取决于工程架构设计，分布式数据库可实现水平扩容。嵌入式数据库（如ChromaDB）QPS有限。
            - **查询精度：** 许多数据库未提供实时精度监测。模型（特别是自训练模型）和数据分布（如大量相似数据）可能导致精度显著下降。VikinDB通过多种内部执行方案和默认参数优化，并提供实时精度监测。
            - **过滤代价：** 过滤比例会显著影响延时和精度。弱Schema设计难以保证过滤效果，可能导致召回量不足或精度劣化。VikinDB支持多种内部执行方案，保证在线过滤不大幅影响性能和精度。
            - **写入性能：** 大规模数据处理场景下，写入QPS和带宽是关键指标。VikinDB支持流式写入和批量导入，可支持日写入百亿向量，并承诺最快速度导入。
            - **写入生效时间：** ANN索引构建成本高，新数据无法立即加入索引。VikinDB通过存算分离架构和旁路化流式更新，可实现2秒内数据可见。
            - **容量限制：** 非分布式数据库容量有限，扩容复杂。云原生数据库（如VikinDB）可提供无限容量扩容。
            - **数据丢失率：** 商业级服务通常提供三副本存储保障数据不丢失。
            - **单个索引规模限制：** 传统ANN索引在数据量大时（如1亿以上），检索延时可能显著退化。云原生架构通过提供更大机器容量，构建更大单体索引，降低计算成本和延时。VikinDB最大支持100GB单体索引。
            - **可靠性保障：** 多可用区、副本承诺、防雪崩能力、弹性伸缩等。
            - **数据加工能力：** 向量数据库需要提供多模态数据（文本、图片、PDF、视频等）的向量化服务，避免用户自行部署推理模型，降低成本和运维复杂度。VikinDB提供内置的向量化服务，支持图片、文本等多种模态。

### [00:51:50 - 00:59:59] 向量数据库应用领域介绍

- **核心论点：** 向量数据库在推荐、广告、搜索、内容安全、AIGC创作和问答等多个领域有广泛应用，尤其在处理非结构化数据和提供语义检索方面具有独特优势。
- **详细展开：**
    - **推荐：** 用户兴趣偏好通过双塔模型转换为user embedding，可推荐内容（视频、文章）转换为content embedding，通过向量相似度计算返回用户感兴趣的内容，进行排序。此外，还可用于多样性打散（过滤同质内容）和重排序（召回和排序模型分离）。
    - **广告：** 与推荐类似，但广告候选规模通常较小，可利用GPU加速检索，追求高精度和高吞吐。
    - **搜索：** 分为站内搜索（如知识库搜索）和全网搜索。VikinDB在这两个场景都有大规模支持。小规模场景关注时效性，大规模场景（百亿以上数据）需考虑资源规划和批量计算（如Spark/Hive导入）。还可支持按时间分片的数据管理，实现数据自动过期。
    - **内容安全/原创保护：** 用于识别搬运、抄袭内容，提供高精度低成本的相似性检索能力。
    - **AIGC创作：** 支持代码Copilot、视频创作、文档助手、游戏角色创建等。通过RAG减少模型幻觉，提高生成质量，支持一键生成周报等应用。
    - **问答：** 针对单一或多个知识库进行问答检索。VikinDB的过滤能力可实现权限隔离，将不同部门或客户的数据放在同一数据库中，通过过滤条件保证用户只能访问有权限的内容。这是RAG短期内难以被大模型记忆能力替代的关键优势。
    - **记忆库：** 探索Agent的本地临时记忆，VikinDB未来将提供标准化解决方案。

### [01:03:50 - 01:19:47] VikinDB向量数据库优化介绍

- **核心论点：** VikinDB通过自研引擎、多索引算法、深度硬件优化、过滤引擎优化和稠密稀疏向量融合检索等技术，实现了高性能、低成本和高精度的向量检索能力。
- **详细展开：**
    - **2.1 特点与差异化**
        - **自研引擎：** VikinDB不依赖Faiss等开源库，从零开始自研向量引擎，支持定制化优化，不受开源社区限制，保证索引格式兼容性。
        - **多索引算法：** 支持HNSW、IVF、DiskANN等主流算法，并针对性优化，性能提升约3倍。还支持一些小众算法。
        - **深度硬件优化：** 与Intel、AMD、Nvidia、华为等硬件厂商合作，利用SIMD指令集（如AVX、AVX512、AMX）和数据连续性优化，提升CPU/GPU检索性能。
        - **过滤引擎优化：** 纯向量检索场景较少，实际应用中常需标量过滤。VikinDB在标量过滤场景下性能可达ES的10倍，并能保证在不同过滤比例（如3%-5%）下无性能退化和精度损失，避免召回量不足或精度劣化。支持行级和列级过滤。
        - **稠密稀疏向量融合检索：** 业界普遍采用RF（Reciprocal Rank Fusion）等混合排序方法，但存在信息损失和系统复杂度问题。VikinDB通过定制的混合ANN索引，将稠密向量和稀疏向量（如关键词权重）融合，实现一次性检索，动态加权，提高召回精度（可提升5-10个百分点），尤其适用于语义不连续的文本（如OCR提取、表格数据）。
        - **向量加密：** 提供Approximate Distance Comparison Preserving Encryption (aDCPE) 技术，在保证检索精度的前提下，对向量进行加密存储，防止数据泄露。即使使用开源模型，通过本地向量化和加密，也可保证数据安全。

### [01:19:47 - 01:28:52] VikinDB大规模云原生架构

- **核心论点：** VikinDB采用大规模云原生架构，通过存算分离、索引构建与服务分离、定制调度和旁路化流式更新等技术，实现了高可用、高弹性、低成本和高性能。
- **详细展开：**
    - **存算分离：** 存储层通过文件存储和宽列存储底座实现多副本和高稳定性，支持无上限容量。
    - **索引构建与服务分离：** 索引构建（如HNSW建图、IVF聚类训练）耗时且计算开销大，甚至高于在线服务。分离后可充分利用异构硬件，构建最大单体索引（VikinDB支持100GB），降低检索成本和延时。
    - **定制构建调度和编排：** 在线集群可部署多个索引，需进行资源隔离、扩缩容约束和控制，防止雪崩和OOM。VikinDB的调度系统能精确预估每个索引的CPU负载和内存占用，确保计算容量充足。
    - **旁路化流式更新：** 新写入数据可不经过完整的索引构建流程，直接写入在线查询索引，实现2秒内数据可见，满足检索型数据库的时效性要求。单分片可支持1000 QPS写入，通过分片可实现更高QPS。
