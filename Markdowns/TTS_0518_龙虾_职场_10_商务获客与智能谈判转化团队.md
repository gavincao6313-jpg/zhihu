## 2. 核心知识字典（Glossary）

- **OpenCloud:** 一个强大的平台，在处理信息流、工作流决策和协作管理方面具有显著优势，是构建多 Agent 自动化团队的基础。
- **Agent (AI智能代理):** 模拟现实世界中的部门、个人或拥有特定技能的角色，能够自主完成特定任务，并在 OpenCloud 框架下进行协作。
- **主 Agent (Main Agent):** 团队中的协调者和最终决策者，负责意图识别、智能调度、流程协调和自动巡检，确保子 Agent 协同工作以实现整体业务目标。
- **子 Agent (Sub-Agent/执行者):** 在主 Agent 协调下执行特定任务的 AI 代理，例如“意向出差 (leadscout)”、“商务谈判 (negotiator)”、“订单执行 (order-executor)”和“策略优化 (strategist)”。
- **Skill (技能):** Agent 具备的特定能力，如 Web Search（网页搜索）、Send Email（发送邮件）、Receive Email（接收邮件）、Feishu Table Write（飞书表格写入）等，通过这些技能 Agent 能够与外部系统交互并执行任务。
- **Identity (身份):** 对 Agent 角色和职责的明确定义，指导 Agent 在执行任务时的行为和决策。
- **Soul (灵魂):** Agent 的核心逻辑和行为准则，包含其任务流程、搜索策略、数据标准化、评分规则、客户画像形成、通知机制等，是 Agent 智能决策的依据。
- **Harbit (心跳/定时任务):** 一种定时执行的任务机制，用于周期性地检查特定条件（如新线索、邮件回复），并触发相应的 Agent 动作，实现自动化流程的持续运行。

### [00:17:41 - 00:49:58] 子 Agent 的创建与配置

- **核心论点：** 逐步创建并配置四个子 Agent (意向出差、商务谈判、订单执行、策略优化)，并为它们定义各自的 Identity 和 Soul，以及所需的工具和数据交互方式。
- **详细展开：**
    - **创建子 Agent：** 在 OpenCloud 终端通过 `agent add` 命令创建四个子 Agent：`lead_count` (意向出差)、`negotiator` (商务谈判)、`order_executor` (订单执行)、`strategist` (策略优化)。所有 Agent 的工作空间目录统一设置为 `Workspace/agents`。
    - **意向出差 (lead_count) Agent 配置：**
        - **Identity：** 定义为“意向出差下的工作空间”，每次打开执行 `lead_count` 的 Soul。
        - **Soul：**
            - 职责：接收主 Agent 传递的线索，利用 `web_search` 能力搜索公司基本信息、联系方式（尤其是邮箱）。
            - 数据标准化：将搜索到的数据进行统一规范。
            - 去重：检查线索池中是否存在重复线索。
            - 评分规则：根据基础分 (0-40)、行为数据 (0-20)、接触机会 (0-20)、历史数据 (0-10) 进行综合评分，形成高、中、低意向评级。
            - 客户画像：基于信息、决策、行为特征评估，形成用户基本画像。
            - 数据落库：将处理后的线索数据写入飞书线索表。
            - 邮箱转化：将获取的邮箱统一格式。
            - 通知：将处理结果通知主 Agent。
        - **工具：** `web_search` (网页搜索)、`feishu_table_write` (飞书表格写入)。
    - **商务谈判 (negotiator) Agent 配置：**
        - **Identity：** 定义为“商务谈判下的工作空间”，每次打开执行 `negotiator` 的 Soul。
        - **Soul：**
            - 职责：接收高意向线索，通过邮件与客户进行多轮沟通、报价、处理异议，并记录谈判日志。
            - 沟通流程：初次接触（发送介绍邮件）、报价（根据客户需求提供方案和价格）、等待回复、异议处理（多轮交互）、确认成交（发送感谢信）。
            - 邮件模板：预设不同阶段的邮件模板和话术。
            - 报价策略：设定多轮报价的折扣和底价保护。
            - 异议处理：预设多种异议（如价格不合适、需要考虑等）的回复话术。
            - 数据落库：将每次沟通内容和谈判结果写入飞书谈判日志表。
        - **工具：** `send_email` (发送邮件)、`receive_email` (接收邮件)、`feishu_table_write` (飞书表格写入)。
        - **通用 Skill：** `send_email` 和 `receive_email` 作为通用技能，在根目录的 `skills` 文件夹下创建，方便所有 Agent 调用。
        - **邮箱配置：** 在 `config` 文件夹下创建 `email_config.json` 文件，配置发件邮箱、授权码（非登录密码，需在邮箱设置中获取）、SMTP/IMAP 服务器和端口。
    - **订单执行 (order_executor) Agent 配置：**
        - **Identity：** 定义为“订单执行下的工作空间”，每次打开执行 `order_executor` 的 Soul。
        - **Soul：**
            - 职责：接收已成交线索，生成定制合同、发送合同、提醒付款、检查付款状态、协调交付。
            - 合同生成：根据沟通内容生成个性化合同（模拟，实际可调用第三方电子签章 API）。
            - 付款检查：模拟检查付款状态（实际可调用银行 API）。
            - 逾期处理：设定逾期付款提醒策略。
            - 数据落库：将订单信息写入飞书订单表。
        - **工具：** `send_email`、`receive_email`、`feishu_table_write`。
    - **策略优化 (strategist) Agent 配置：**
        - **Identity：** 定义为“策略优化下的工作空间”，每次打开执行 `strategist` 的 Soul。
        - **Soul：**
            - 职责：定期分析成交数据、失败数据、谈判数据、报价数据等，生成优化建议，反哺前序 Agent 的工作。
            - 优化建议：根据数据分析结果给出具体的优化建议。
        - **工具：** `feishu_table_read` (飞书表格读取)、`feishu_table_write`、`data_analysis` (数据分析)。
    - **主 Agent (main) 配置：**
        - **Identity：** 定义为“商务团队的协调者”，负责协调四个子 Agent。

## 4. 遗留问题与下一步行动（如有）

- **关键词识别优化：** 在实际演示中，关键词识别（如“下单”与“购买”）可能存在不准确性，需要根据实际业务场景对 Agent 的 Soul 中的关键词进行更精细的配置和优化。
- **第三方 API 集成：**
    - **电子签章：** 视频中合同生成为模拟文本，实际应用中可集成如“e签宝”等第三方电子签章服务的 API，实现合同的自动化生成和签署。
    - **银行/支付系统：** 订单付款状态的检查可集成银行或支付平台的 API，实现自动化的款项核对。
    - **物流系统：** 对于涉及实物交付的业务，可集成第三方物流系统的 API，实现订单的物流跟踪和状态更新。这需要为 Agent 编写新的 Skill 来调用这些 API，并可能需要考虑物流信息推送机制（而非定时查询）。
- **数据可视化与报表：** 策略优化 Agent 提到了数据分析，但视频中未展示具体的数据可视化和报表功能。未来可考虑集成 BI 工具或在飞书表格中构建更复杂的视图，以便更直观地展示获客和谈判效果。
- **多用户/多 OpenCloud 实例支持：** 视频中演示的是单用户单表场景，实际业务中可能需要支持多用户、多表或多 OpenCloud 实例协同工作，这需要更复杂的权限管理和数据同步机制。
- **持续优化与迭代：** 策略优化 Agent 的核心在于反哺前序工作。需要建立一套机制，定期收集 Agent 运行数据，分析效果，并根据分析结果迭代 Agent 的 Soul 和 Skill，以持续提升自动化团队的效率和智能水平。
