## 2. 核心知识字典（Glossary）

-   **BI (Business Intelligence)**: [00:07:45] 商业智能的简称，主要用于帮助企业分析和可视化自身或外部数据，以了解业务、市场和运营情况，从而支持决策。它通常是一个技术、平台、产品或技术集合，主要处理结构化数据，输出形式包括报告、报表、数据预测和挖掘结果。
-   **对话式BI (ChatBI)**: [00:17:35] 一种基于自然语言交互的智能化查询系统。用户通过自然语言与系统对话，系统自动获取数据并生成报表，旨在提高数据获取和分析效率，实现实时响应，并对非技术用户更加友好。
-   **NL2SQL (Natural Language to SQL)**: [01:32:32] 将自然语言（Natural Language, NL）转化为结构化查询语言（SQL）的过程，是ChatBI的核心技术之一，包括Schema Linking、知识检索、SQL生成和SQL自校正等步骤。
-   **Schema Linking**: [01:43:43] NL2SQL过程中的一个关键步骤，旨在将自然语言查询中的实体（如表名、字段名、值）映射到数据库的相应Schema元素。其目的是从庞大的数据库Schema中筛选出与用户查询最相关的部分，以减少大模型处理的上下文长度，提高SQL生成准确性。
-   **多轮对话改写 (Multi-turn Dialogue Rewriting)**: [00:36:07] ChatBI中的一项技术，用于将用户在多轮对话中带有指代和省略的上下文信息，整合改写成一个完整、独立的查询（query），以消除歧义，使后续模型阶段能专注于处理一个语义完整的query。
-   **RAG (Retrieval-Augmented Generation)**: [00:33:23] 一种结合检索和生成的技术，在ChatBI中用于知识检索和Schema Linking。通过从外部知识库或数据库中检索相关信息，并将其作为上下文提供给大语言模型，以增强模型的回答能力和准确性，尤其适用于领域特定知识和模糊匹配。
-   **COT (Chain of Thought)**: [00:25:34] 思维链，一种提示工程技术，通过要求大语言模型输出其解决问题的中间逻辑步骤，来提高其在复杂任务（如SQL生成）上的推理能力和准确性。

### [00:07:45 - 00:12:12] BI的背景与价值

-   **核心论点：** 商业智能（BI）是帮助企业分析和可视化数据以支持决策的技术集合，其核心价值在于解决数据孤岛、提升展示效率、辅助决策和预测挖掘。
-   **详细展开：**
    -   **什么是BI**: Business Intelligence的简称，主要用于帮助企业分析和可视化自身或外部数据，以了解业务、市场和运营情况。BI通常是一个技术、平台、产品或技术集合，最常见的输出是报告、报表或数据预测挖掘结果。BI主要处理结构化数据，帮助我们通过分析和快速查看结构化数据内容来做决策。
    -   **BI提供的核心价值（总结为4点）**:
        1.  **解决数据孤岛问题**: 例如零售行业，各门店有独立的销售、库存、客户购买行为数据。BI能整合这些分散数据，帮助企业更有效地管理库存，减少缺货或过剩。
        2.  **提升数据展示效率**: 例如金融行业，实时展示股票股价、指数走势等交易数据，通过图表和仪表盘方式直观呈现，帮助用户实时了解数据情况。
        3.  **辅助决策管理**: 例如工业领域，通过分析生产线运行时间、故障率，发现生产效率下降的原因（如机器老化、维护不当），从而辅助管理层做出判断。
        4.  **预测挖掘**: 通过分析历史数据，挖掘有价值信息。例如零售业在节假日前备货，BI可根据历史数据预测特定商品在即将到来的节假日所需备货量，提升店铺运营效率。

### [00:27:09 - 00:35:25] 对话式BI的落地挑战

-   **核心论点：** 对话式BI在实际落地中面临四大挑战：数据库设计、数据库内容标准化、数据描述和领域知识管理，这些都需要细致的前期准备和策略。
-   **详细展开：**
    -   **1. 数据库设计**:
        -   应使用清晰、规范的数据库作为查询数据源。
        -   真实工业场景中数据库常存在杂乱、数据重复（如设备故障导致多记录）、数据不完整（如维护人员变动导致空值）、Schema不一致（如新旧传感器数据字段不兼容）等问题。
        -   这些问题需要通过数据清洗、ETL流程等预处理，将数据整合到清晰规范的数据表（如数据仓库）中，再接入ChatBI系统。
    -   **2. 数据库内容标准化**:
        -   数据表应明确业务的最小粒度，这对于后续聚合操作至关重要。
        -   例如，生产指标表可能以“工厂-产线-小时”为最小粒度，明确每个小时的生产效率、开机次数、产量等。这决定了SQL中聚合函数（如SUM、AVG）和GROUP BY的写法。
    -   **3. 数据描述**:
        -   数据库的表和字段应有明确的描述（comment），包括字段解释（中英文对照）、是否为枚举值及其具体值、字段单位（如KL、KG）。
        -   这些描述对于模型理解数据含义、生成正确SQL（如单位转换）和处理枚举值（如“假班”、“乙班”或地点缩写“bj”对应“北京”）至关重要。
    -   **4. 领域知识**:
        -   领域知识（如字段别名、业务公式、特定值映射、默认查询范围等）对ChatBI至关重要。
        -   应以自然语言形式存储这些知识，并通过RAG（检索增强生成）方式在需要时检索使用。
        -   **示例**:
            -   **字段别名**: 如金融领域的“ROE”是“股东权益回报率”的简称。
            -   **业务公式**: 如工业领域的“开机率 = 开机成功次数 / 开机总次数”。
            -   **值映射**: 如“女性”在数据库中可能存储为“F”（female）。
            -   **默认查询范围/聚合方式**: 如查询“三月份哪个工厂的产量最高”，默认产量可能需要求和（SUM），但用户可能需要平均值（AVG），这些默认行为也应作为知识存储。
            -   **领域特定逻辑**: 如医疗领域中“血细胞比容水平超过正常范围”的具体数值定义，这些高度专业的知识需要明确提供给模型。

### [00:50:02 - 01:16:23] ChatBI核心技术：NL2SQL

-   **核心论点：** NL2SQL是将自然语言转化为SQL查询语言的过程，其基本工作流程包括Schema Linking（值、字段、表检索与排序）、知识检索、SQL生成和SQL自校正，旨在从海量数据中精准生成SQL。
-   **详细展开：**
    -   **NL2SQL基本工作流程**:
        1.  **Schema Linking**: 将自然语言映射到数据库的表、字段、值。
        2.  **知识检索**: 检索出自然语言查询所需的知识。
        3.  **SQL生成**: 生成相应的SQL查询语句。
        4.  **SQL执行与SQL self-correction**: 执行生成的SQL语句，如果出现错误，则反馈给模型进行自我校正。
    -   **Schema Linking（基于Query的结构化数据检索）**:
        -   **检索过程**: 分为召回和排序两个阶段，以提高检索表现。
        -   **1. 值表示与检索**:
            -   **索引构建**: 尽量不将数值类或时间类的值加入索引，主要加入枚举类的值。
            -   **召回**: 输入query，通过BM25关键词索引和向量索引（将query embedding化后进行相似度搜索），从存储的值内容中召回候选值。
            -   **排序**:
                -   **基于向量的排序**: 输入`query + [候选值所在表名] + [候选值所在表描述] + [候选值所在字段名] + [候选值所在字段描述] + [候选值]`，通过计算相似度得分进行排序，筛选出Top-K或高于阈值的候选值。
                -   **基于LLM的排序**: 使用Prompt模板，将query和召回的候选值信息（包括表名、表描述、字段名、字段描述、候选值）提供给LLM，让LLM判断哪些值是完成SQL所需的。
            -   **解决的问题**:
                -   **隐式表达**: 许多值在query中未明确提及其所属的表和字段。例如，“青岛3月份的生产效率情况”中，“青岛”是工厂名而非地区名，通过值检索可匹配到正确的“工厂”字段。
                -   **模糊匹配**: 用户查询可能使用简称或缩写（如“工行”），通过值检索可匹配到数据库中存储的全称（如“中国工商银行股份有限公司”）。
        -   **2. 字段表示与检索**:
            -   **召回**: 如果单表字段数不超过50个可省略召回。输入`query + [值]`，通过BM25关键词索引和向量索引（存储字段名+字段描述），召回候选字段。
            -   **排序**:
                -   **基于向量的排序**: 输入`query + [召回字段所在的表名] + [召回字段所在的表描述] + [召回字段名] + [召回字段包含的值]`，进行排序。
                -   **基于LLM的排序**: 使用Prompt模板，将query和召回的候选字段信息提供给LLM，让LLM判断哪些字段是完成SQL所需的。
            -   **解决的问题**:
                -   **显式声明的字段**: 直接匹配query中明确提及的字段。
                -   **隐式声明的字段**: 例如，“今年营收最高的工厂”中，“今年”隐式指代时间列，“营收”隐式指代“营业收入”列。通过字段检索，可匹配到正确的时间和营业收入字段。
                -   **字段的模糊匹配**: 例如，“营收”可模糊匹配到“营业收入”字段。
        -   **3. 表表示与检索**:
            -   **召回**: 如果数据库表数不超过50个可省略召回。输入`query + [检索回来的字段名] + [字段描述] + [检索回来的值]`，通过BM25关键词索引和向量索引（存储表名称+表描述），召回候选表。
            -   **排序**:
                -   **基于向量的排序**: 输入`query + [候选表名] + [候选表描述] + [候选表包含的字段名] + [候选表包含的值]`，进行排序。
                -   **基于LLM的排序**: 使用Prompt模板，将query和召回的候选表信息提供给LLM，让LLM判断哪些表是完成SQL所需的。
            -   **解决的问题**:
                -   **隐式声明的表**: 用户查询通常不直接提及表名（如“利润表”），而是通过字段和值间接指代。例如，“今年营收最高的工厂”需要通过“工厂”和“营收”字段找到对应的表。
                -   **多表匹配**: 排序阶段可筛选出多张相关表，支持多表Join查询。例如，查询工厂营收可能需要“工厂ID表”和“指标表”进行关联。
    -   **知识检索**:
        -   **召回**: 如果知识条目数不超过50个可省略召回。输入`query + [知识文本]`，通过BM25关键词索引和向量索引（存储知识文本），召回相关知识。
        -   **排序**:
            -   **基于向量的排序**: 输入`query + [知识文本]`，进行排序。
            -   **基于LLM的排序**: 使用Prompt模板，将query和召回的知识文本提供给LLM，让LLM判断哪些知识是完成SQL所需的。
        -   **知识检索解决的问题**:
            -   **专业术语别名**: 如“工行今年的ROE是多少？”需要检索“ROE = Return on Equity (股东权益回报率)”的知识。
            -   **值映射**: 如“工行开户的女性客户有多少？”需要检索“女性 = F”的知识。
            -   **业务逻辑**: 如“3月份哪个工厂的产量最高？”需要检索“产量group by月份时默认以平均为聚合方式”的知识。
            -   **领域特定规则**: 如医疗领域中“血细胞比容水平超过正常范围”的具体数值定义。
    -   **SQL生成**:
        -   **基于Schema Linking结果，结合Query生成SQL**:
            -   **Prompt设计**:
                -   **仅Instruction/Query**: 最简单的方式，直接给指令和问题，模型难以写对。
                -   **加入简单SQL表示**: 提供数据库Schema信息（如`CREATE TABLE Highschooler (ID, name, grade)`），模型能更好地理解表结构。
                -   **加入SQL示例数据**: 提供`SELECT * FROM Highschooler LIMIT 3`的示例数据，帮助模型理解数据存储格式（如名字大小写、年级表示方式），提高准确性。
                -   **使用CREATE TABLE语句表示Schema**: 这种方式在模型训练中更常见，稳定性更好，能完备展示外键等信息。
                -   **加入检索到的Comment和知识**: 将Schema Linking和知识检索的结果（包括字段描述、枚举值、单位、领域知识等）整合到Prompt中，进一步提高模型准确性。
                -   **Few-shot示例**: 提供多个问题-SQL对作为示例，引导模型生成正确SQL。
                -   **COT (Chain of Thought)**: 要求模型输出逻辑链（如“确定查询的表”、“确定查询的字段”、“确定筛选条件”、“编写查询语句”），提高复杂SQL的准确性。
        -   **多步生成SQL**:
            -   **分类与分解Prompt**: 首先对query进行分类（EASY, NON-NEST, NEST），根据分类结果使用不同的Prompt和Few-shot示例。
                -   **EASY**: 单表查询，无需嵌套或Join。
                -   **NON-NEST**: 多表Join查询，无需嵌套。
                -   **NEST**: 需要嵌套查询。
            -   **示例**:
                -   **NEST类型**: “找出移动计算课程的先修课程的课程标题是什么？”模型会先生成一个子查询来找到先修课程ID，再用外层查询找到标题。
    -   **SQL执行与SQL self-correction**:
        -   执行生成的SQL，如果出现错误，将错误信息（如“Limit 1”写成“Limiter1”的语法错误）和原始SQL反馈给大模型，让模型进行自我校正，生成修正后的SQL。

### [01:21:47 - 01:28:11] ChatBI核心技术：数据可视化 (Data Visualization)

-   **核心论点：** 数据可视化将查询结果转化为直观的视觉格式（图表、图形），以简化数据理解，其核心在于利用大模型的代码生成能力，结合预定义的绘图工具，实现灵活、定制化的图表生成。
-   **详细展开：**
    -   **目的**: 将数据转化为视觉格式（如图表、图形、地图），简化数据理解，帮助用户快速发现数据中的模式和洞察。
    -   **常见图表类型**:
        -   **条形图/柱状图**: 比较不同类别的数量。
        -   **折线图**: 展示数据随时间变化的趋势。
        -   **散点图**: 比较两个变量之间的关系。
        -   **饼图**: 展示部分占整体的比例。
        -   **热力图/地图**: 通过颜色变化展示数据密度或强度。
    -   **绘图方式**:
        -   通常使用Python的matplotlib、seaborn等绘图库，也可以使用内部支持自定义图表。
        -   核心是利用大模型写代码的能力，而非直接生成图片。通过生成调用绘图工具的函数代码，可以实现高度定制化和灵活的图表生成，适应不同客户的需求（如3D/2D、多层图等）。
    -   **核心问题**:
        1.  **多字段映射**: 数据中存在多个字段，如何根据用户意图选择合适的字段（横轴、纵轴、图例等）和图表类型（趋势图、占比图等）。
        2.  **客户定制化**: 客户有各自定制化的图表需求，如何更好地集成。通过生成调用工具的代码，可以灵活适应。
        3.  **图表文本信息**: 除了数据本身，还需要生成图表的标题、图例等文本信息。
    -   **Data Visualization Workflow (Prompt示例)**:
        -   **Prompt结构**: 分为两大部分：Tool的定义和输入。

-   **详细展开：**
    -   **大模型选择**:
        -   可以直接调用商用大模型API（如GPT-4, GPT-4o），这些模型在推理和代码生成方面表现优秀。
        -   如果考虑成本、QPS或性能，可以对小参数量开源模型（如千问72B或Code系列模型）进行微调，使其达到商用模型的水平，从而降低运营成本。
    -   **多轮对话性能**:
        -   多轮对话改写会将历史对话压缩成一个完整的query，后续处理量和耗时与单个query类似。
        -   但如果query非常复杂，需要几百行SQL才能解决，目前BI工具或大模型的能力边界可能无法很好处理，通常仍需数据分析师介入。
    -   **NL2SQL准确率**:
        -   在讲师团队的实际落地场景中，对于Easy类型的简单SQL（数据清晰、无歧义、无复杂业务逻辑），准确率可达95%以上。
        -   对于嵌套（NEST）类型的复杂SQL，准确率相对较低，约80%左右。
    -   **数据权限**:
        -   数据权限控制通常在两个层面：ChatBI系统层面（控制用户对数据库、字段的访问权限）和数据库本身的用户权限。
        -   更多依赖标准BI系统提供的权限管理功能，对用户可查询的表和字段进行限制，超出权限则不返回数据或提示错误。
    -   **召回机制**:
        -   “召回”是检索领域专业名词，指从海量数据（如几十万到亿级别的值）中快速（秒级或毫秒级）检索出与query最相关或所需的内容。
        -   检索通常分为召回和排序两阶段。召回阶段追求速度，使用相对简单的方法（如BM25、向量索引），确保相关内容被包含在召回结果中（例如从500个候选召回中包含所需的5个）。排序阶段则使用更复杂的方法（如LLM或向量相似度）将最相关的Top-K结果排到最前面。
    -   **跨数据库操作**:
        -   一般建议将相关业务领域的数据表整合到同一个数据库中，以简化操作。
        -   技术上可以实现跨数据库查询，但通常需要在外部增加一层数据库的召回和筛选逻辑，且从数据库设计角度不推荐频繁进行跨数据库查询。
    -   **Schema Linking中有效表和字段的判断**:
        -   通过排序逻辑实现。可以使用大模型直接判断召回的候选表/字段中哪些是有效的，或者通过计算query与表/字段embedding的余弦相似度，筛选出高于预值的表/字段。
        -   最终通过层层筛选，决定哪些表和字段会被大模型用于SQL生成。
    -   **基础数据构建**:
        -   构建清晰的数据库表定义、字段定义、专业术语等是整个BI领域的前置核心工作量。
        -   这包括数据清洗、数据融合（ETL），将不同数据源、不同格式的数据整合到统一的数据仓库中。通常有专门的数据治理项目来完成这些工作，为上层BI查询系统（无论是ChatBI还是传统BI）提供高质量的数据基础。
    -   **模型对不同维度聚合的支持**:
        -   如果提供最细粒度的数据（如小时或分钟级），模型通常能很好地进行时间维度（月、年、趋势）和地区维度（如果地区字段存在）的聚合。
        -   但如果聚合维度在数据库中不存在（如只存了城市名，但要按省聚合），则模型无法完成。
    -   **值检索的含义**:
        -   指数据库中具体存储的数据内容。例如，工厂字段中存储的“青岛工厂”、“北京工厂”或“工厂1号”等。
    -   **NL2SQL大模型微调数据设计**:
        -   训练数据与推理数据结构相似。每条样本包括：
            -   **Instruction部分**: 表定义、表描述、字段定义、字段描述、值检索结果和内容、知识、COT指令。
            -   **Output部分**: 具体的COT逻辑和最终生成的SQL结果。
        -   训练数据来源：可以通过人工标注，但更常见的是利用现有大模型（如GPT-4）生成query和对应的中间步骤/SQL作为训练数据。验证集和测试集则通过人工编写query和SQL来确保质量。
    -   **模糊匹配与隐式表达的区别**:
        -   **模糊匹配**: 指用户使用的词语是数据库中实际值的缩写或简称（如“工行”对应“中国工商银行股份有限公司”）。
        -   **隐式表达**: 指用户query中未明确提及某个Schema元素（如表名、字段名），但通过上下文或值可以推断出来。例如，“青岛的生产效率怎么样”中，“工厂”字段是隐式表达。
