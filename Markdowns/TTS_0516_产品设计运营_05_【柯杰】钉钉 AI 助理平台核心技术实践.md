## 2. 核心知识字典（Glossary）

- **AI助理 (Agent)：** 钉钉平台对智能体的称呼，强调其拟人化、辅助用户完成任务的特性。它具备人设、技能和知识，旨在降低用户理解和使用门槛，弥补传统全代码和低代码在某些行业覆盖不足的问题。
- **Grounding：** 指将自然语言中的实体（如人名、时间、地点）准确地映射到业务系统中的具体ID或数据格式的过程。这是实现自然语言交互与业务系统集成的关键技术，确保AI助理能够理解并操作实际业务数据。
- **Stream协议：** 钉钉AI助理平台采用的一种基于WebSocket的全双工连接协议，用于应用服务器与钉钉服务器之间的数据传输。它解决了传统Webhook模式下内网穿透、HTTPS证书管理、加解密签名等复杂问题，提升了开发效率和安全性。
- **Planning (规划推理模块)：** AI助理的核心模块，负责理解用户意图、规划执行步骤、调用相应技能、并进行反思和结果观察。钉钉平台通过持续升级Planning模块，借鉴自动驾驶理念，旨在提升AI助理的整体体验和效率。
- **Agentic Workflow：** 指由LLM（大语言模型）自动推理和编排的工作流，而非传统的人工编排。演讲者认为这是未来开发模式的重要变革方向，能够大幅降低业务流程定制的门槛，让普通用户也能通过自然语言定制自己的工作流。

### [00:18:13 - 00:25:30] LUI与GUI的融合：实用性与确定性优先

- **核心论点：** 为了提升AI应用的实用性和确定性，钉钉强调LUI（Language UI）与GUI（Graphical UI）的融合，通过交互式卡片和中断处理机制，确保用户操作的可控性和准确性。
- **详细展开：**
    - **实用性优先：** AI应用的关键在于能否被用户每天使用。一个场景跑通Demo容易，但让目标用户每天使用是巨大挑战。因此，确定性至关重要。
    - **LUI与GUI融合：**
        - **场景示例：** 用户通过LUI（自然语言）请求“创建一个日程”。
        - **问题：** 创建日程、删除/修改数据等高敏感操作，若完全依赖大模型推理，可能因不确定性导致误操作。
        - **解决方案：** 引入GUI（交互式卡片）。在LUI交互后，系统弹出交互式卡片（GUI），让用户确认日程详情、修改时间等，再点击“新建日程”按钮执行。
        - **优势：** 用户多一步确认，提升了操作的可控性和确定性，避免大模型不确定性带来的严重后果。
    - **中断处理机制：**
        - **借鉴操作系统：** AI助理的推理过程是一个主流程。当需要用户确认时，会发生“中断”，当前推理过程被挂起。
        - **流程：** 1. 发生中断 (操作确认) -> 2. 中断处理 (用户在GUI卡片上操作) -> 3. 返回主流程 -> 4. 继续推理。
        - **类比：** 类似于操作系统的中断处理，以及早期嵌入式开发中的内存管理（如分页式内存管理），这些成熟的计算机科学概念可借鉴到Agent开发中。
    - **Token管理：** 大模型上下文长度有限，即使是128K的Context，在复杂React推理任务中，若全部用满，注意力会分散，效果不佳。实际可用Token数有限，需要像操作系统内存管理一样进行优化。
    - **技能与交互方式：** 技能除了返回JSON，让大模型自动生成Markdown展示外，还可以调用API发送自定义卡片，实现更精美的排版和交互能力（如按钮回调事件）。
    - **多模态交互：** 除了文本和卡片，还支持图表、音视频展示、图文混排、语音播报，未来甚至可探索数字人等多样化交互形式。

### [00:25:30 - 00:32:56] 技能的打磨：以日程为例 - 确定性来自正确认知

- **核心论点：** 技能的质量比数量更重要。实现高质量技能的关键在于对大模型能力边界的正确认知（MPF），区分大模型擅长推理不擅长计算的特点，并结合工程化手段解决复杂问题。
- **详细展开：**
    - **质量优先：** 不稳定、效果不佳的技能再多也无意义，应聚焦技能质量。
    - **确定性来自正确认知 (MPF - Model Product Fit)：**
        - **理解大模型能力：** 开发者需清晰了解大模型在何种场景下表现良好，何种场景下存在不足。
        - **工程化与大模型结合：** 当下大模型能力仍有不足，复杂场景仍需大量工程化开发。关键在于识别哪些环节适合大模型推理，哪些环节需要工程化计算。
        - **案例：** 大模型擅长推理，不擅长计算。例如，即使是GPT-4o在比较9.11和9.8大小这种小学数学问题上仍可能出错。因此，计算逻辑应由工程代码处理，确保100%准确性。
        - **意图识别：** 大模型在意图识别方面进步显著（如通义千问），目前意图识别不准多是由于技能描述不够清晰，而非模型本身问题。多个技能描述放在一起时，容易出现模糊，需要像辨别真假币一样进行对比优化。

### [00:32:56 - 00:42:56] 技能的打磨：以日程为例 - 人物地点时间处理

- **核心论点：** 日程管理涉及人物、地点、时间等核心要素，这些要素的准确识别和转换（Grounding）是AI助理落地的关键挑战。钉钉平台将这些复杂能力沉淀到框架层面，通过声明式API简化技能开发。
- **详细展开：**
    - **核心要素：** 人物、地点、时间是许多技能（如日程、待办）的共同要素，处理这些要素的复杂性是技能开发的主要挑战。
    - **Grounding的必要性：**
        - **自然语言到业务ID的转换：** 例如，用户说“给张三发邮件”，大模型能识别“张三”，但系统需要的是张三的邮箱ID。在钉钉系统中，创建日程/待办需要用户ID，而非汉字名称。
        - **OpenAI GPTs的局限：** GPTs无法直接识别“张三”为邮箱地址，需要用户提供完整邮箱。
        - **钉钉的解决方案：** 将自然语言中的人名、时间等转换为业务系统可识别的ID（如UnionID、Staff ID、工号），这一转换过程称为Grounding。
    - **复杂场景举例：**
        - **人物识别：**
            - 多人对话中加人、修改会议时间/地点、补充不认识的人的描述。

    - **意图识别与参数提取：** 意图识别由基础大模型（如通义千问）实现，参数提取由Agent框架的Planning模块完成，而非专用Agent。
    - **开发与部署：**
        - 钉钉AI助理平台提供模板中心，可快速搭建助理。
        - 技能开发支持Python和Java，有详细文档和教程。
        - 手机和PC端均可操作，PC端功能更全。
        - 默认标准版免费，高级功能收费。
        - 可发布到微信公众号、小程序、网页，也可通过SDK/API集成到自有App，实现脱离钉钉平台使用。
    - **主动性：** 目前AI助理主动性较弱，主要通过定时任务实现主动汇报。未来将增强环境感知能力，让AI助理能感知事件变化并自动执行任务。
    - **数据与隐私：**
        - 历史聊天记录不会全部作为上下文，会有压缩和限制。
        - 企业内部AI助理涉及的隐私数据不会泄露，采用多租户架构。
        - 用户打分数据主要用于收集反馈，不会直接用于模型训练，避免随意打分影响效果。
    - **多模态能力：** AI助理具备多模态理解能力，可处理图片、语音等复杂信息，例如工单助理可理解截图中的问题。
    - **Agentic Workflow：** 钉钉未来将提供面向Agentic Workflow的产品能力，通过LLM自动推理生成工作流，而非人工编排。
    - **WebSocket优势：** 在AI应用中，WebSocket的全双工流式交互（如语音对话、长语音转文字）比SSE的单向流式交互更具优势。

    - **AI在传统制造：** 在传统制造领域，AI在质检环节有重要应用，可辅助提效。
    - **多AI助理设计：** 设计多个AI助理是为了提高专注力，每个助理专注于特定领域，避免技能过多导致准确性下降。
