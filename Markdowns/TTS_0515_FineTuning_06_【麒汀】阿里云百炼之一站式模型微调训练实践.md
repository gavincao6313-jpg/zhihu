## 2. 核心知识字典（Glossary）

-   **SFT (Supervised Fine-Tuning / 有监督微调):** 一种大模型微调方法，通过使用有标注的指令数据（Promote-Completion对）来训练模型，使其学习特定任务或领域的能力，是目前业务落地中最常用的微调方式。SFT又分为全参SFT（更新模型所有参数）和PEFT（参数高效微调，如LoRA、QLoRA，只更新部分参数）。
-   **LoRA (Low-Rank Adaptation):** 一种参数高效微调（PEFT）方法，通过在预训练模型中注入小的、可训练的低秩矩阵来更新模型参数，显著减少了训练所需的计算资源和内存，同时保持或提升模型性能。
-   **QLoRA (Quantized LoRA):** 在LoRA的基础上进一步优化，通过将预训练模型量化到4比特精度加载到GPU内存中，并使用分页优化器处理内存峰值，从而进一步降低GPU内存消耗，使得在更小的GPU上也能进行大模型微调。
-   **Prompt Engineering (提示工程):** 一种不涉及模型参数更新的优化方法，通过精心设计输入给大模型的提示（Prompt），引导模型生成期望的输出。它是使用大模型时成本最低、最便捷的优化手段，尤其适用于利用大模型的启发式学习（In-context Learning）能力。
-   **Continue Pre-train (继续预训练 / CT):** 在现有基础大模型上，使用大量无标注的领域数据进行进一步的预训练，以使模型学习新的领域知识。这种方法需要大量数据和计算资源，成本较高，适用于需要模型学习大量新知识的场景。
-   **RLHF (Reinforcement Learning from Human Feedback / 基于人类反馈的强化学习):** 一种高级的大模型训练方法，通过收集人类对模型输出的偏好反馈，训练一个奖励模型，然后利用强化学习算法优化大模型，使其输出更符合人类偏好。该方法技术难度高，对数据和训练过程要求严格，但能显著提升模型与人类对齐的能力。

### [00:10:14 - 00:18:51] 大模型微调（SFT）的背景与现状

-   **核心论点：** SFT是目前大模型业务落地最常用的微调方式，其核心在于平衡训练效率、选择合适的训练手法（全参、LoRA、QLoRA）以及有效的评测机制。
-   **详细展开：**
    *   **SFT的普遍性：** 大模型微调通常包括基模训练、基模二次增强（Post Continual Pre-train）、SFT和RLHF。在实际业务落地中，SFT（有监督微调）是最常用的方法。
    *   **训练效率考量：**
        *   降低训练数据构建成本。
        *   降低训练计算资源成本，缩短训练周期。
        *   业界普遍认为，训练数据数量的指数增长，模型效果呈现线性增长，因此存在“最佳实践”的数据量。
    *   **训练手法选择：**
        *   **全参SFT (Full Finetuning):** 在调优时更新模型内部的整体参数，覆盖面广，但消耗内存大，需要高性能GPU。
        *   **LoRA (PEFT):** 参数高效微调，主要更新模型中一部分层的矩阵参数（可能只占总参数的1%），显著减少GPU内存消耗和所需卡数。
        *   **QLoRA:** 在LoRA基础上进一步优化，模型加载为4比特，相比16比特有4倍内存差异，可在更小的GPU上运行，适应国内GPU资源稀缺的现状。
    *   **模型评测：**
        *   **训练过程：** 根据`Train loss`和`Validation loss`曲线判断模型拟合程度，Loss值越接近0越好。
        *   **训练完成：** 结合评测工具进行评估。两种手段：后台自动运行评测脚本；部署模型服务后，通过调用API或调试窗口体验。阿里云百炼目前倾向于后者，提供更灵活的评测方式。
    *   **阿里云百炼的支持：**
        *   已支持全参SFT和LoRA微调。
        *   提供超过十几个超参数，目前线上版本主要支持“循环次数 (Epoch)”、“批次大小 (Batch Size)”、“学习率 (Learning Rate)”。春节前将推出更多高阶参数。
        *   训练过程中会透明化展示Loss曲线和预估训练时长。
        *   正在开发预训练（CT）和RLHF，SFT数据量小、成本低，CT和RLHF数据量大、成本高，且RLHF对效果影响大，难度也高。

### [00:18:51 - 00:25:35] 大模型训练基本理念与方法

-   **核心论点：** 大模型训练路径包括无监督预训练（基模）、指令微调（SFT）和强化学习（RLHF），用户应从Prompt Engineering开始，逐步深入到SFT和继续预训练，以平衡效果和成本。
-   **详细展开：**
    *   **大模型生产路径：**
        *   **Base Pre-train LLM (基础预训练大模型):** 无监督方式，不需要标注数据，但需要高质量文本。这是所有基模的来源。
        *   **Instruction Tuned LLM (指令微调大模型):** 基于指令数据进行Fine-tune，通常由大模型厂商提供，数据标注非常宝贵。
        *   **Dedicated LLM (专属模型):** 针对特定领域，基于无标注数据进行继续预训练（Continue Pre-train），或基于有标注指令数据进行Fine-tune。
    *   **Transformer架构与微调原理：**
        *   Transformer架构包含Encoder和Decoder，将用户输入（如“我有一只猫”）编码成词向量，再通过Decoder生成输出（如“Avocado”），这是一个文本翻译的例子。
        *   **Pre-training (预训练):** 输入是无格式的长文本，模型通过学习更新参数。
        *   **Fine-tuning (SFT):** 输入是成对的、有格式的数据，如`Promote`和`Completion`。
    *   **SFT数据格式示例：** 采用JSONL格式，每行一个JSON对象，包含`messages`数组，每个元素有`role` (system, user, assistant) 和 `content`。`System`定义模型角色，`User`是Promote，`Assistant`是Completion。

### [00:25:35 - 00:30:54] 大模型训练方法对比与选择

-   **核心论点：** 不同的训练方法在数据规模、算力消耗、训练时间、成本和适用场景上存在显著差异，用户应优先考虑Prompt Engineering，再逐步深入到SFT和继续预训练。
-   **详细展开：**
    *   **Prompt Engineering (提示工程):**
        *   **数据规模：** 极少/无。
        *   **训练模式：** ICL/范式式训练（In-context Learning）。
        *   **算力大小：** 少（仅推理消耗）。
        *   **训练时间：** 0。
        *   **成本：** 低/无。
        *   **适用场景：** 利用基础大模型能力，通过提示词和少量示例进行优化。
        *   **优势：** 最优先使用，成本最低，调试最方便。OpenAI也分享过Prompt构建方法，强调清晰、背景描述、期望输出。
    *   **微调训练 (Supervised Fine-tune / SFT):**
        *   **数据规模：** 少（几百条分类任务数据即可）。
        *   **训练模式：** 1到N训练/效果训练。
        *   **算力大小：** 中/小。
        *   **训练时间：** 中（几小时到几天）。
        *   **成本：** 中/低。
        *   **适用场景：** 针对特定领域/特定任务进行优化，解决大模型在特定领域回答不准确的问题，学习少量新知识。
    *   **继续预训练 (Continue Pre-train / CT):**
        *   **数据规模：** 大（数十亿Token）。
        *   **训练模式：** 0到1训练/领域训练。
        *   **算力大小：** 大（几十张卡）。
        *   **训练时间：** 非常长。
        *   **成本：** 高。
        *   **适用场景：** 从0到1训练大模型，或针对特别特殊领域学习大量新知识。
    *   **RLHF (基于人类反馈的强化学习):**
        *   **数据规模：** 大。
        *   **训练模式：** 强化学习联合训练。
        *   **算力大小：** 大。
        *   **训练时间：** 中长。
        *   **成本：** 中高。
        *   **适用场景：** 利用人类反馈使模型输出与人类偏好对齐，技术能力要求强，训练复杂。
    *   **业务应用路径建议：** 从Prompt Engineering开始，若效果不佳再尝试SFT，若仍无法解决领域知识问题，则考虑Continue Pre-train。这是一个自顶向下的优化过程，也符合成本控制的原则。

### [00:41:31 - 00:46:10] 一些反例：数据可能带来的Bad Case

-   **核心论点：** 不良的训练数据会导致模型出现重复输出、学习错误文本风格、无法理解复杂指令等问题，因此数据质量至关重要。
-   **详细展开：**
    *   **1. 重复性输出 (Repetitive Output Pattern):**
        *   **问题：** 训练数据中包含重复的回答模式，导致模型在推理时也重复输出。
        *   **示例：** 用户问“人工智能在医学上会如何应用？”，训练数据中Assistant的回答多次出现“例如，AI可以帮助医生更准确地诊断疾病，并为病人提供更有效的治疗方案。”
        *   **影响：** 模型会快速学习这种重复模式，导致输出质量低下。
    *   **2. 不合理的语法表达 (Unreasonable Grammatical Expression):**
        *   **问题：** 训练数据中存在不规范或不自然的语言表达，导致模型学习到错误的文本风格。
        *   **示例：** 用户问“有什么方法可以帮我提高心情？”，Assistant回答“帮助提升心情的方法有很多...”，其中“提高心情”不如“提升心情”常见。
        *   **影响：** 模型可能无法准确命中用户意图，或产生不自然的回复。
    *   **3. 简单的问题无法激发复杂能力 (Simple Questions Fail to Elicit Complex Responses):**
        *   **问题：** 训练数据中的Promote过于简单，无法充分激发大模型的复杂理解和回答能力。
        *   **示例：** 用户问“这些方法有什么好处？”，Assistant回答“这些方法可以帮助您减轻压力，改善情绪...”。如果Promote没有提供足够的上下文，模型可能无法给出有深度的回答。
        *   **影响：** 模型无法展现其全部能力，输出可能过于泛泛。
