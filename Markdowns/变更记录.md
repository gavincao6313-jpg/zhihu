## 主题:Gemini 配额、路由与多模型轮询(2026-06-16 ~ 06-17)

**起因:** WIN 报告 Gemini 分窗严重浪费 API 调用,日产出从 ~20 跌到 3-4。逐层排查后,真实根因最终定位为**免费层每个 flash 模型只有 20 RPD**,而代码只用单一模型被卡死。

下面按时间顺序记录每一步改动。

---

### 变更 1 — `GEMINI_MAX_FRAMES` 200 → 700  【已提交 57762ff】

- **根因:** 200 帧 ≈ 52k token(每帧 320px JPEG≈258 token),远低于任何真实上限(250k TPM / 1M 上下文)。正常 1–2.5h 的 Gemini 视频被误判为超限,拆成 6+ 窗口,每窗 `1+MAX_CONTINUATIONS=3` 次调用 + assembly,耗尽配额。原 commit 还误判 007(1092帧)为"1M 上下文溢出",实际仅 ~282k token,真正越限的是 250k TPM。
- **改动:** `GEMINI_MAX_FRAMES` 常量 200→700(TPM 安全阈值,700帧≈180k 图像 token,单次调用守在 250k TPM 内),注释改写。
- **验证:** `py_compile` + 窗口数模拟(≤700→单次,007→2窗)。
- **文件:** `batch_process_external.py`

### 变更 2 — 删 Gemini 分窗死代码 + 超护栏 reroute Qwen  【已提交 57762ff】

- **根因(经 WIN `.progress_batch.json` 核实):** 路由纯按时长(`route_provider`:≤2.5h→gemini,>2.5h→qwen)。唯一 reroute 是 **Gemini→Qwen**(retry-failed + failed_stage==gemini + >2.5h),**不存在 Qwen→Gemini**。所有 >700 帧视频(706–1241)都 >2.5h → 走 Qwen;Gemini 只见 ≤2.5h 视频(实测最大 658 帧)。故 `_process_gemini_windowed` 在 174 个 Gemini 视频中 **0 例触发 = 死代码**。注:007 是 Qwen 视频,不可当 Gemini 分窗案例。
- **决议:** 「保留安全网 + 删死代码」。
- **改动:** 删除 `_process_gemini_windowed` 函数 + `_GEMINI_WINDOW_ASSEMBLY_PROMPT` 常量;`process_single_video` 中 provider==gemini 且 `frames>GEMINI_MAX_FRAMES` 改为 reroute 到 `_process_qwen`(复用 Qwen 250 帧分窗,作为防单次 Gemini 调用 >250k TPM 的护栏);`qwen_client is None` 时安全失败 `failed_stage=gemini_frames_over_guard`,不崩溃。`_est_calls_for` 超护栏分支改按 Qwen 分窗估算。
- **验证:** `py_compile` + 路由冒烟 + 无残留死符号。
- **文件:** `batch_process_external.py`

### 变更 3 — 配额计费归属 + 调用前闸门预判  【已提交 57762ff】

- **问题:** reroute 把 gemini 视频实际交给 Qwen,但计费仍记到 `gemini_calls`;且调用前的 Gemini 配额闸门会误拦本该 reroute 的视频。
- **改动:** reroute 时 `result["provider"]="qwen"`,`process_batch` 用 `billed_provider=result.get("provider",provider)` 计入 `qwen_calls` 并存 `entry["provider"]`;闸门加护栏预判——`entry` 已存 `frame_count>GEMINI_MAX_FRAMES` 的视频提前按 Qwen 过闸(首次无 frame_count 仍保守按 Gemini,由 Phase 3 内部 reroute 兜底)。
- **文件:** `batch_process_external.py`

### 变更 4 — 外部 AI 审查回炉(两轮 APPROVE)  【已提交 57762ff】

- **发现1[高]:** `_est_calls_for` 对 `provider==qwen`/`will_reroute` 提前 `return 3`,大 Qwen 视频(含 reroute 后存为 qwen 的)走不到分窗估算 → 预算低估(fc=1000 估 3 实 15)。**修复:** 统一 `on_qwen` 判定,按 `ceil(fc/QWEN_MAX_FRAMES)` 估窗口 + assembly。
- **发现2[高]:** 达 `--max-calls-per-run` 时在 `save_batch_progress` **之前** break,已烧调用不落盘 → 下次重复烧 API。**修复:** `should_stop` 延迟 break,先存盘+计数+打印再停。
- **发现4[低]:** 注释残留"007 window"、reroute 后失败日志误写 GEMINI。**修复:** 注释改 reroute-Qwen 表述;失败日志用 `result.get("provider")`。
- **发现3[中]:** 首次无 frame_count 时 reroute 受 Gemini gate 限制。**决议:暂不处理(已知限制)**——死锁仅在 Gemini 无 key/配额永久满 + 出现 >700帧≤2.5h 视频时成立,当前 0 例。
- **验证:** `py_compile` + 路由/预算/配额归属/闸门 分层冒烟;两轮第三方审查 APPROVE WITH CAVEAT。

### 变更 5 — 免费配额认知重大更正  【文档已改,未提交】

- **真相(AI Studio rate-limit 页实测截图,2026-06-17):** 免费层**每个 flash 模型的 RPD 上限 = 20**,且 **per-model 独立计**(RPM 5 flash / 10 lite,TPM 250k)。`CLAUDE.md` 里的"250 RPD"是**过时错误数据**,曾连续误导分析两轮。官方文档已不再印固定免费 RPD,一切以 AI Studio 为准。
- **关键结论:** `DEFAULT_DAILY_LIMIT_GEMINI=20` 是**精确匹配单模型真实配额**,不可调高(撞 429)。RPD 扩容正解 = **多 flash 模型轮询**(每模型独立 20 RPD),而非付费 Qwen(Gemini 免费 / Qwen DashScope 付费)。
- **改动:** 更正 `CLAUDE.md` 的 Gemini 配额表(250→实测 20,标注 per-model + 多模型轮询);`cerebrum.md` 同步记录(此为本文件启用前的最后一次 cerebrum 写入)。
- **可用免费 flash 文本模型(各 20 RPD):** `gemini-3.5-flash` / `gemini-2.5-flash` / `gemini-3-flash` / `gemini-2.5-flash-lite`。当前 AI Studio 峰值:3.5-flash 已打爆(29/20),其余 3 个几乎闲置。

### 变更 6 — Gemini 多 flash 模型轮询(免费 RPD 扩容)  【待外部审查 → 提交】

- **目标:** 把日产能从单模型 20 扩到 **4 模型 × 18 = 72/天**,全免费、质量同级,消化几百积压。
- **改动(`batch_process_external.py`):**
  1. `DEFAULT_DAILY_LIMIT_GEMINI` 20→**18**(语义改为 **PER-MODEL** 日上限,留 2 防 429)。
  2. 新增 `GEMINI_MODEL_POOL`:默认 4 模型(质量降序 `gemini-3.5-flash`→`gemini-2.5-flash`→`gemini-3-flash`→`gemini-2.5-flash-lite`);**显式设 `GEMINI_MODEL` 环境变量时池只含该单模型**(兼容 WIN 手动止血);`GEMINI_MODEL_POOL` 环境变量可自定义。
  3. 新增 `_pick_gemini_model(daily, per_model_limit)`:按 `daily['gemini_calls_by_model']` 分模型计数,选未满的最高质量模型,全满返回 None。
  4. `_process_gemini(..., model=)` / `process_single_video(..., gemini_model=)`:接收并传递选中模型。
  5. `process_batch` gate:轮询选模型,全满则跳过该视频;计数按选中模型分别累加(保留 `gemini_calls` 总和向后兼容)。
  6. `dry_run` / 每视频 status / final summary / argparse help:显示模型池、per-model 用量、总容量、按 4×18 估天数。
- **验证:** `py_compile` ✓;端到端轮询冒烟(per_model_limit=2):`3.5-flash×2→2.5-flash×2→3-flash×2→lite×2→停`,分模型计数准确;`_pick_gemini_model` 单元冒烟通过。
- **风险:** 核心 L2/L3(配额预算逻辑);`gitnexus detect_changes` 评 **CRITICAL / 21 touched**,但多数是常量区增行导致的**位置偏移误报**(QWEN_MAX_FRAMES / MAX_AUDIT_PROBES / load_batch_progress / save_batch_progress / _find_clean_frames_binary 未改逻辑);affected_processes 全在 batch 脚本内,未跨核心模块。
- **审查包:** `Markdowns/审查包-Gemini多模型轮询.md` + `审查包-Gemini多模型轮询.diff`(+63/-25,已落盘)。
- **状态:** ⏳ 待外部 AI 审查 → 修复(如有)→ 提交 → WIN `git pull` + `sync_to_zhihu_file.bat`。
- **未验证项:** 未跑 lint(本机无 ruff);Mac 无真实视频,轮询未在真实 Gemini API 下端到端跑(需 WIN);`gemini-3-flash`/`gemini-2.5-flash-lite` 的确切 API model id 待真实调用确认。
