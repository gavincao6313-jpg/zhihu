## 2. 核心知识字典（Glossary）
- **Agent (代理)**: 在 OpenCloud 平台中，Agent 是一个独立的智能实体，可以被赋予特定的职责、配置不同的模型、工作空间和权限，以执行特定任务。
- **多 Agent 模式**: 相对于单一 Agent 模式，多 Agent 模式通过创建多个具有不同职责和配置的 Agent，并让它们协同工作，以处理更复杂、多样化的任务场景。它解决了单一 Agent 上下文窗口限制、提示词污染和串行工作模式的缺点。
- **调度层 (Scheduler Agent)**: 在多 Agent 团队中，调度层是一个特殊的 Agent，负责接收用户请求，并根据请求内容智能地将任务分发给最合适的子 Agent 进行处理。
- **Skill (技能)**: 在 OpenCloud 中，Skill 是 Agent 具备的特定能力，通常通过 Markdown 文件定义，包含工具链、描述、触发条件和执行逻辑，使 Agent 能够执行文件操作、数据处理、API 调用等具体任务。
- **Bunnies (绑定配置)**: OpenCloud 中用于配置 Agent 与特定渠道（如飞书群、个人用户）之间绑定关系的机制，允许为不同的群组或用户指定不同的 Agent 来提供服务。

### [00:00:49 - 00:04:05] 单一 Agent 的局限性
- **核心论点：** 单一 Agent 在处理复杂、多样化任务时存在上下文窗口限制、提示词污染、串行工作模式和权限风险等问题，因此需要引入多 Agent 模式。
- **详细展开：**
    1.  **上下文窗口被填满**: 单一 Agent 需处理所有任务，导致其完成任务所需的历史信息和背景知识（上下文）过多，会读取大量 Markdown 文件（如 `Worker Space` 中的 `Makadam` 文件、`User.md`）并写入大量 `Memory`，长时间运行后，Agent 回答问题会变得不精准。虽然可以通过 `/New` 开启新对话临时解决，但仍会读取不相关的 `Memory` 文件。
    2.  **提示词污染**: 复杂任务需要详细的背景信息和指令，如果所有任务都由单一 Agent 处理，会导致提示词变得冗长和复杂，降低效率和生成质量。
    3.  **串行工作模式**: 单一 Agent 只能一个接一个地执行任务（例如，既写代码又搜集资料），是串行工作模式。虽然可以通过子代理实现并行，但子代理任务是临时的，不具备持久性，且无法独立配置权限和模型。
    4.  **权限风险**: 单一 Agent 难以实现权限隔离。例如，一个在群聊中回答问题的 Agent 不应拥有本地文件读写权限，以防任何人修改本地文件。
- **视觉/屏幕内容：** 幻灯片标题 "为什么单 Agent 会出问题？"，列出以下四点：
    - 上下文窗口被填满
    - 提示词污染
    - 串行工作模式
    - 权限风险
- **重要金句/原话：** "如果所有的事情只让单一的 Agent 来去进行的话，就会出现上下文窗口被填满。"

### [00:04:05 - 00:07:49] 多 Agent 模式的优势与拆分原则
- **核心论点：** 多 Agent 模式通过团队协作、权限隔离、模型配置灵活性和专注度提升，有效解决了单一 Agent 的问题，并提供了何时拆分 Agent 的指导原则。
- **详细展开：**
    1.  **多 Agent 模式的优势**:
        *   **权限隔离**: 可以为不同的 Agent 配置不同的读写和执行命令权限，以及安全检查，大大降低整体风险。例如，群聊 Agent 不应有本地文件读写权限。
        *   **模型配置灵活性**: 针对不同任务配置不同模型。复杂代码任务可交由更昂贵的复杂模型，简单任务则使用快速且便宜的模型。
        *   **专注度提升**: 每个 Agent 只关注自己的上下文，拥有独立的 `User` 文件和 `Memory` 文件，从而提高任务的整体效率。
    2.  **Agent 拆分原则**:
        *   **何时拆分**: 当每个 Agent 需要不同的记忆和专业知识，并且其工作流程完全独立，不依赖其他 Agent 时，应进行拆分。例如，搜集资料、写文章和检查合约安全性是完全不同的场景，需要不同的背景信息。
        *   **何时不拆分**: 如果只是同一项工作有不同的模板，或者任务属于同一个上下文（如前端加后端开发一个工具），则没必要拆分。
- **视觉/屏幕内容：**
    - 幻灯片标题 "多 Agents 模式优势"，列出四点：权限隔离、模型配置、专注度提升、团队协作。
