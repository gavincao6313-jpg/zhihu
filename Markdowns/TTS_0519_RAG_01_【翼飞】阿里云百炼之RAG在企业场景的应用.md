## 2. 核心知识字典（Glossary）
- **RAG (Retrieval Augmented Generation) 检索增强生成：** 一种大模型应用范式，通过在生成前进行检索，将相关信息作为上下文提供给大模型，以解决大模型知识时效性、幻觉等问题。它在视频中被定义为“解决一些实时性的问题，前面加个搜索”。
- **Chunk (切块/片段)：** 将长文档切分成小块文本单元，以便于存储、检索和作为大模型上下文输入。视频中提到“把文档等文本的，比如说500个字或什么的，切成一小块”。
- **DocGraph (文档图谱)：** 一种将文档内容细化为树形或图形结构的表示方法，旨在捕捉文档内部的层级关系、父子关系和并行关系，以优化检索和问答效果。视频中解释为“将我们的文档细化成这么一个树形或者图形的一个图”。
- **幻觉 (Hallucination)：** 大模型在生成回复时，额外增加文档中没有的信息，导致答案不准确或虚构的现象。视频中举例说明了刘德华专辑中添加“忘情水”的错误。
- **百炼 (bailian.aliyun.com)：** 阿里巴巴的企业大模型平台，提供基于通义千问及其他开源大模型的应用构建、模型训练、知识管理等能力，特别强调其在企业级文档问答方面的RAG能力建设。

### [00:10:00 - 00:18:28] RAG系统流程与挑战
- **核心论点：** RAG系统通过数据准备、切块、索引、检索和Prompt构建，增强大模型在文档问答中的能力，但面临文档预处理复杂、知识召回效果要求高、大模型生成可解释性差等挑战。
- **详细展开：**
    - **RAG第一步：数据准备。** 包括PDF、Word、表格、图片等多种格式文档。由于大模型Prompt长度限制，文档需切块（chunk）。
    - **传统搜索与向量搜索：**
        - 传统搜索：通过关键词召回，如BM25等文本算法。
        - 向量搜索（Vector Search）：解决语义近似问题，例如“住哪”和“地址”在字面上不相似但语义相关。
    - **索引与检索：** 将切块后的文本通过关键词或语义向量存入数据库。当用户提问时，从数据库中检索相关chunk。
    - **Prompt构建：** 检索到的chunk会拼接到大模型Prompt中，通常包含三部分：
        - System Prompt（黑色）：定义助手角色、知识领域、回答风格等。
        - 检索内容（蓝色）：检索回来的相关chunk。
        - 用户Query（黄色）：用户提出的问题。
    - **RAG系统三大难点：**
        1. **文档预处理：** 文档种类多（PDF, Word, Table, Picture），结构复杂，切块易碎，难以保障语境完整性。
        2. **知识片段召回：** 召回效果要求高，容易搜不回来，对最终效果影响大。
        3. **大模型生成可解释性：** 大模型是生成式模型，输出原因未知。需要提供引用（如Bing Search的“杠12345”）。
    - **百炼的解决方案：**
        - **知识预处理：** 针对PDF等文档进行OCR、版面分析、知识管理（可视化）。
        - **模型优化：** 传统大模型不适合文档问答，百炼提供可解释性（引用）能力。
        - **召回增强：** 结合关联词搜索和语义搜索。
        - **一键式上传：** 支持上传PDF、图片、PPT等多种格式，通过OCR等技术解析。

### [00:18:29 - 00:22:02] RAG与长文本处理的必要性及效果提升
- **核心论点：** 尽管大模型上下文长度不断增加，但直接将长文本全部输入大模型的效果并不理想，RAG通过检索和重排序能显著提升长文本问答的准确率。
- **详细展开：**
    - **大模型长文本处理的局限性：** 引用最新研究，即使GPT-4等模型支持128K Token，当答案位于文本深处（如40%以后）时，准确率会显著下降。直接将《哈利波特》全书放入大模型，准确率仅约75.8%。
    - **RAG的工作原理：**
        1. 文档切块（chunk），存储在数据库或向量索引中。
        2. 用户Query转换为向量和关键词。
        3. 从数据库中检索相关chunk（例如Top 5或Top 10）。
        4. 将检索到的chunk拼接到大模型Prompt前，生成回复。
    - **RAG效果提升路径：**
        1. 基础RAG（仅余弦相似度召回）：准确率约75.8%。
        2. 使用OpenAI的Ada-002 Embedding：效果显著提升。
        3. 综合关键词和语义向量：进一步提升。
        4. 召回50个片段后进行重排序：再次提升效果。
    - **结论：** RAG相比直接使用大模型处理长文本，效果有巨大提升。

### [00:26:16 - 00:28:53] 难点1：多模态数据表示
- **核心论点：** 企业文档中包含文本、表格、图片等多种模态数据，传统方案难以统一处理。百炼通过Data2Text方法将不同模态数据统一转化为文本描述，实现多模态统一搜索。
- **详细展开：**
    - **问题：** 企业文档通常包含文本、表格、图片。传统方案对每种模态分别建模，生成各自的语义向量进行搜索，但图片无法直接用关键词搜索，且不同模态的向量难以直接融合。
    - **人类处理方式：** 人类会自然地将文字、表格、图片转化为自己理解的文字描述（如摘要、描述）。
    - **百炼的解决方案：Data2Text方法。**
        1. **预训练与微调：** 利用业界多模态预训练模型，微调出Data2Text模型。
        2. **模态转换：** 将文本做摘要，图片生成描述，表格生成描述，统一转化为文本。
        3. **统一搜索：** 通过文本方式进行多模态搜索。
        4. **排序与生成：** 最终通过排序生成回复。
    - **成果：** 该方法在产品上实现了多模态输入输出能力，并在学术界公共榜单上取得良好效果，相关工作已发表论文。

### [00:28:57 - 00:36:02] 难点2：索引怎么做 (DocGraph)
- **核心论点：** 传统线性切分文档容易截断语义，忽略文档的结构和层级关系。百炼提出DocGraph方法，将文档转化为图谱结构，并通过构建对话数据集，实现多轮次交互式问答，提升索引和问答的准确性。
- **详细展开：**
    - **传统索引方法的局限性：**
        1. **语义截断：** 线性切分（如按固定字数）可能将产品名等关键信息截断，导致召回失败。
        2. **忽略结构：** 文档通常有标题、模块、父子层级、同级并列等结构关系，传统方法无法利用。
        3. **多轮交互需求：** 人类理解复杂文档需要多轮交互才能确定答案，例如判断仓鼠性别需要澄清是成年还是未成年。
    - **百炼的解决方案：DocGraph。**
        1. **层级切分：** 利用DoCR模块对文档进行层级切分。
        2. **对话创建：** 根据层级结构，人工标注生成对话数据，模拟多轮交互。
        3. **数据集特点：** 标注了6000个对话，超过10万个Turn，涵盖政府、保险、医疗、公共等5个领域。
    - **DocGraph核心技术：**
        - **DocGraph概念：** 将文档细化为树形或图形结构，捕捉层级、父子、并行关系，类似于知识图谱。例如，人才引进政策包含城市、申请条件、流程等层级。
        - **规则文本与表格抽象：** 针对规则文本（如受理条件、数量限制、禁止性要求）和表格，抽象成DocGraph结构。例如，申请条件与办理流程的关系，以及多条件（Conjunction）的满足要求。
        - **多样化问句类型：** 支持是否类、多次确认类、反问类等多种问句类型，以适应多轮交互。
    - **成果：** 该数据集和方法在EMNLP22被接收，并举办了ACL2023的Doc2Dial Workshop比赛，吸引了近400个队伍和15篇论文投稿。

### [00:36:08 - 00:44:11] 难点3：大模型生成效果保障

        - **百炼的解决方案：构建评测Agent。**
            - **借鉴多人评判标准：** 利用大模型本身能力构建评测Agent。

        - **成果：** 该工作已发表论文。
