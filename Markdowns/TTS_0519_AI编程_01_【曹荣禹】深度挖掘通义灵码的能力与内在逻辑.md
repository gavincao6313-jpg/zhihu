## 2. 核心知识字典（Glossary）
- **通义灵码 (TONGYI Lingma):** 阿里巴巴通义实验室推出的一款智能编码助手，旨在通过AI技术提升开发效率，提供代码生成、补全、智能问答、AI程序开发等功能。
- **代码生成补全 (Code Generation and Completion):** 根据用户当前编写的代码上下文，AI模型自动预测并推荐后续可能需要编写的代码片段，包括行级、块级、函数级甚至类级的代码。
- **AI程序开发 (AI Program Development):** 指AI程序员（通义灵码的一个产品）根据用户提出的自然语言需求，自主规划、分析代码仓库、生成修改建议并执行代码变更，以实现功能开发或缺陷修复。
- **RAG (Retrieval-Augmented Generation):** 检索增强生成，一种结合了信息检索和生成模型的技术。在代码领域，它通过从代码库中检索相似代码片段作为参考，辅助大模型生成更准确、无幻觉的代码。
- **Code-VBench:** 通义灵码团队提出的一个站在用户角度的代码补全评测数据集，旨在更真实地评估代码大模型在多种复杂补全场景下的性能，并采用单测通过率作为核心评估指标。
- **Lingma-SWEGPT:** 通义灵码团队训练的一个专门用于解决SWE-bench数据集上项目需求开发和缺陷修复任务的模型，它结合了工具使用（Agent）和大型语言模型的能力。

### [00:18:27 - 00:55:59] 编码助手中代码生成补全的探索
- **核心论点：** 代码生成补全是AI编码助手最刚需的场景，面临多语言、复杂上下文和庞大三方库的挑战。通义灵码通过训练专属代码大模型、采用Fill-in-the-Middle模式和整库感知等技术，并提出了Code-VBench评测集，以解决这些问题并提升用户体验。
- **详细展开：**
  - **代码生成补全的定义与挑战：**
    - **定义：** 根据上下文，补全当前行中“代码缺失”的部分，包括行级和块级代码。
    - **刚需场景：** AI编码助手最核心功能之一。
    - **困难与挑战：**
      - **多语言支持：** 需支持Java, Python, Go, JavaScript等多种语言，存在长尾问题，每种语言有不同框架、规范，知识密度高。
      - **复杂上下文：** 不仅需考虑当前文件，还要理解项目中其他文件定义的类和函数，避免幻觉；需学习庞大的二方库和三方库，不同版本特性各异，学习困难。
  - **业界解决方案：代码专属大模型与RAG：**
    - **代码专属大模型：**
      - 训练思路：基于基础语言模型（如Llama, Qwen, GLM, Mistral）训练代码模型（如Code Llama, Code Qwen, CodeGeex, CodeMistral）。
      - 数据：使用GitHub, Gitee等网站的海量真实代码仓库数据，学习各种语言、框架、二方/三方包。
      - 训练模式：采用**Fill-in-the-Middle (FIM)** 模式，模型需同时考虑光标上文和下文来补全中间代码，区别于传统从前往后生成。
      - **整库感知：** 组织数据时考虑整个项目库，引入其他文件定义的类和函数信息，避免模型生成幻觉。
    - **超视距项目级别代码理解（基于RAG的代码补全生成）：**
      - **本文件：** 丰富上下文场景（for循环、if判断、函数、类），多样生成粒度，建模用户可能行为（如只生成函数签名），理解长文件中的函数定义。
      - **跨文件：** 解析项目中其他文件的类/函数签名，概括文件内容，构造跨文件调用语句学习样例。
        - **示例：** 演示`DWCRerportServiceImplement.java`中调用`DWCReport.java`中定义的类和字段，模型能准确补全，无幻觉。
      - **本仓库（RAG）：** 对仓库文件进行多粒度切分，向量化存储（基于embedding模型），根据用户补全位置构造合理检索请求，从向量库检索相似代码片段，并将其作为提示给补全模型，模型学习参考相似代码生成。
      - **三方包：** 解析仓库依赖的第三方包及其版本信息，构造数据让模型学习。
    - **通义灵码的6点可感知能力（已实现4点，规划2点）：**
      - 已具备：补全位置可感知、项目中调用文件可感知、相似代码可感知、三方包依赖可感知。
      - 规划中：数据结构可感知（感知对象所属类、类中函数/字段）、执行环境可感知（感知项目安装环境）。
  - **站在用户角度提出新的代码补全评测集 (Code-VBench)：**
    - **为什么要提出新评测集：**
      - 现有评测集（如HumanEval, ClassEval, EvalCodeBench）多基于函数签名生成函数体，与真实代码补全场景相距甚远。
      - 真实代码补全场景丰富：补全if块、函数参数、完整函数、完整类等，现有数据集无法涵盖。
      - 需要能反映模型在丰富场景下生成质量和用户行为的评测集。
    - **Code-VBench特点：**
      - **真实业务驱动：** 从GitHub爬取近期高Star仓库作为测试项目，涵盖各种框架，更贴近真实业务。
      - **丰富补全场景：** 通过Tree-sitter解析工具对文件进行AST解析，识别for循环、if语句、函数、单行语句等代码块，构造多样化的补全场景。
      - **准确评估体系：** 基于单测进行评测。筛选有单测的GitHub仓库，通过小工具分析并安装环境依赖，运行单测来判断模型补全代码的正确性（单测通过率）。相比相似度计算更客观。
    - **数据集构建过程：**
      - **1. 数据爬取与单测构建：** 爬取GitHub仓库，识别并运行其自带的单测（如`test_sorting.py`测试`sorting.py`中的`bubble_sort`和`quick_sort`）。通过迭代请求大模型解决环境依赖报错，确保单测可正常运行。
      - **2. 贝侧函数识别：** 通过Tree-sitter解析单测函数，识别其对应的贝侧函数（如`test_bubble_sort`对应`sorting.py:bubble_sort`）。
      - **3. 代码块解析与上下文构造：** 对贝侧函数进行AST解析，将其拆分为多个可补全的代码块（如for循环、if判断、单行语句）。根据不同代码块构造上下文和待补全内容，模拟用户真实补全场景。
      - **4. 场景丰富性分析：** 统计阿里云内部Python代码开发场景，发现补全内容多样（普通语句、判断循环、异常块、注释、注解等），补全粒度多样（单行、多行、行内、空补全）。评测集按此比例构造。
    - **评测结果：**
      - 构造了四个主要场景：完整代码块补全、上下文不完整补全、代码块内部补全、引入RAG相似代码片段补全。
      - 在主流语言模型（GPT-4, Claude-3, Deepseek, Mistral, Qwen）和代码大模型（CodeLlama, DeepseekCoder, CodeQwen, CodeGeex, CodeMistral）上进行评估。
      - 结果显示代码大模型在代码补全能力上优于通用语言模型。
      - **发现模型问题：**
        - **无法停止：** 模型在补全完所需内容后，继续生成多余代码，对用户造成干扰。Code-VBench能检测出此问题。
        - **错误跨越多个代码块：** 模型未能理解当前代码块已完整，继续预测下一个代码块的内容，导致错误。
