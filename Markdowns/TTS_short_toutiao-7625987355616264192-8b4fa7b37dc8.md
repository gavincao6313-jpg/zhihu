# 把Obsidian当IDE用！拆解Karpathy三层架构，零成本打造个人百科

## 1. 内容概览
本视频介绍了Andrej Karpathy提出的“AI驱动知识库”工作流，旨在解决传统RAG（检索增强生成）系统无状态、无法沉淀新洞察的问题。核心思路是将笔记软件（如Obsidian）视为源代码库，利用终端Agent（如Claude Code）作为程序员，通过类似CI/CD的自动化流程，实现知识的自动编译、重构和版本管理。视频详细阐述了该方案的三个步骤：使用本地数据确保隐私、利用具备文件读写能力的Agent进行维护、以及通过规则文件约束Agent行为。最终目标是让知识库从静态存储变为动态增长的“活体”系统。

## 2. 时间线

### [00:00:00 - 00:01:00] RAG的痛点与Karpathy的新思路
视频开篇指出传统个人RAG方案本质是只读（read-only）的无状态系统，对话产生的新洞察无法留存。引用Karpathy在GitHub分享的工作流，提出将知识库变成代码库的概念。类比解释：Obsidian是IDE和Markdown源代码，Claude Code是7x24小时在线的程序员，生成的Wiki是编译产物。整个流程如同全自动的CI/CD流水线。

### [00:01:00 - 00:02:00] Agent如何自动维护知识库
详细描述Agent的工作机制：录入新闻文档后，Agent自动提炼重点、重构Wiki页面、打理双向链接。若发现内容冲突，会抛出Warning；定期运行脚本检查死链、补全概念、标记过时API（类似单元测试）。提问产生的高质量对比表会自动Commit归档回Wiki，使查询不仅是Read，也是Write。提到Karpathy的实验：在100篇来源、40万字的知识库中，抛弃VectorDB和RAG Pipeline，直接让模型读目录找文件，结果精准且架构干净。

### [00:02:00 - 00:02:58] 实施三步走与核心理念
总结落地方法：1. 前端使用免费纯本地数据（如Obsidian），数据在硬盘上更踏实；2. 执行Agent首选Claude Code或Cursor/Windsurf，需具备读写本地文件能力；3. 使用规则文件（Rules file）约束Agent，例如规定文件夹不可动、命名规范等。强调大模型时代的最佳开发体验是让AI帮忙维护知识底座，人类腾出脑子做判断。

## 3. 关键事实
*   **核心问题**：传统RAG是无状态的HTTP请求式系统，无法积累增量洞察。
*   **解决方案**：基于Karpathy的工作流，利用LLM Agent对本地知识库进行“编译”和维护。
*   **角色类比**：
    *   Obsidian (Markdown文件) = 源代码/IDE
    *   Claude Code (Agent) = 程序员
    *   Wiki/整理后的内容 = 编译产物
*   **技术优势**：利用现代终端Agent（如Claude Code）成熟的跨文件重构能力，替代传统的向量数据库检索。
*   **实验数据**：Karpathy在100篇文档、40万字规模的知识库中验证了该方法的有效性，无需VectorDB。
*   **实施工具**：
    *   笔记软件：Obsidian
    *   Agent工具：Claude Code, Cursor, Windsurf
    *   约束机制：根目录下的规则文件（Rules file）

## 4. 可检索细节
*   **Karpathy工作流**：一种将LLM Agent与本地笔记软件结合，实现知识库自动更新和关联的方法。
*   **CI/CD类比**：将知识录入视为“Compile”，Agent自动处理链接和冲突视为“CI测试”，归档视为“CD部署”。
*   **Prompt策略**：不要求Agent逐行敲代码，而是丢给Agent让其搭建脚手架（Scaffolding）。
*   **具体维护动作**：
    *   提炼重点
    *   重构Wiki页面
    *   打理双向链接
    *   抛出Warning（内容冲突时）
    *   清理死链接
    *   补全未写完的概念
    *   标记过时API（@tag）
*   **推荐Agent**：Claude Code（首推）、Cursor、Windsurf。关键在于能读写本地文件。
