## 2. 核心知识字典（Glossary）

- **智能体 (Agent):** 视频中指代能够自主行动、与外部环境（包括互联网、其他智能体）进行交互的AI实体。它将取代现有软件，为个人和企业提供服务。
- **协议 (Protocol):** 定义系统之间通信规则的规范，包括数据格式、交换流程和角色约定。在智能体语境下，特指智能体之间进行高效、标准化交互的规则。
- **MCP (Model Context Protocol):** 模型上下文协议，主要解决大语言模型（LLM）与外部数据和工具的集成问题，被视为构建智能体的技术。
- **A2A (Agent to Agent):** 一种智能体协议，主要设计用于解决企业内部智能体之间的复杂协作问题，通过任务分包模式进行交互。
- **ANP (Agent Network Protocol):** 智能体网络协议，旨在成为智能体互联网时代的HTTP，定义智能体之间的连接方式，构建开放、安全、高效的协作网络，强调AI原生、去中心化身份和P2P架构。
- **DID (Decentralized Identifiers):** 去中心化身份标识符，W3C提出的一种规范，ANP基于此设计了智能体身份方案，允许智能体拥有自己的身份，无需在每个平台注册账号即可进行互联互通。

- **详细展开：**
    - **什么是协议？**
        - 定义了系统之间通信的规则，包括数据格式、交换流程和角色约定。例如HTTP。
    - **AI或智能体为什么需要协议？**
        - **技术路线对比：**
            - **API:**
                - 优点：效率高。
                - 缺点：无标准，开发成本高，国内API稀缺。
            - **Computer Use (如Cloud):**
                - 优点：供给多（AI可学习人使用电脑/手机）。

                - 缺点：无法访问APP，安全性问题（云端虚拟机操作Web需提供用户名密码）。
            - **Protocol (AI原生连接方式):**
                - 优点：效率最高，标准化好。

### [00:22:14 - 01:08:45] MCP、A2A、ANP协议详解

- **核心论点：** 详细介绍MCP、A2A、ANP各自的设计目标、架构、核心概念和工作原理，为后续对比奠定基础。
- **详细展开：**
    - **MCP (Model Context Protocol) 是什么？**
        - **目标：** 解决LLM与外部数据和工具的集成问题。
        - **整体架构：** MCP Host (AI应用) -> MCP Module -> MCP Protocol -> MCP Server -> 现有应用 (Google Docs, DB, Web API)。
        - **核心概念 (5个):**
            - `Root`: 客户端向服务端提供访问本地文件和LLM能力。
            - `Sampling`: 让MCP Server使用客户端的LLM能力。
            - `Prompt`: 服务器告诉客户端如何使用资源/工具的提示词。
            - `Resource`: 服务器向客户端公开资源（如Google文档）。
            - `Tools`: 服务器可以自信模型调用的工具（如HTTP命令、高德地图API）。
            - **实际应用：** Resource和Tools使用最多。
        - **核心流程：** CS架构，客户端主动连接服务端。
            1. **初始化：** 协商MCP版本和各自能力。
            2. **操作：** 客户端拉取/调用服务端工具或资源。
            3. **关闭：** 终止连接。
        - **传输层：**
            - **标准IO:** 同一进程内直接通信。
            - **HTTP SSE:** 通过网络HTTP协议连接。
        - **协议格式：** 主要是JSON文档，使用JSON-RPC进行远程调用。
        - **身份：** 基于OAuth 2.1设计，支持客户端动态注册。
        - **典型应用场景：** Claude Chatbot、AI IDE、Blender等通过MCP接入应用或工具。
        - **技术意义：**
            1. 提供标准化集成接口，促进生态发展。
            2. 增强模型能力。
            3. 推动AI原生连接协议技术路线。
        - **面临挑战 (LangChain CEO和LangGraph Lead的公开讨论):**
            1. **身份验证：** 分布式环境下解决身份认证困难，OAuth 2.1并非最佳方案（解决资源授权而非系统间身份认证）。
            2. **复杂性：** 提示词客户端、文件与Sampling、服务器日志等过于复杂。
            3. **服务器扩展性：** 需要无状态协议。
            4. **协议架构：** CS架构，服务端无法主动触达客户端。
    - **A2A (Agent to Agent) 是什么？**
        - **发布时间：** 2024年4月10日。
        - **架构：** 两个Agent之间通过A2A Protocol连接。A2A将MCP视为构建智能体的技术。
        - **工作原理：** 基于任务分包模式。用户通过本地智能体，经A2A协议连接远端智能体，远端智能体可再连接其他智能体，共同完成任务。
            - **任务分包：** 大任务拆分小任务，由不同远端智能体处理。
            - **自然语言描述任务。**
            - **关键技术：**
                - **智能体描述：** 使用Agent Card（智能体身份证）描述智能体能力。
                - **智能体发现：** 与ANP使用相同技术。
        - **工作流程：**
            1. 启动任务。
            2. 代理发现（中心化注册/目录发现）全网智能体。
            3. 能力查询（智能体A问智能体B能做什么）。
            4. 智能体A将任务交给智能体B。
            5. 任务执行，A和B持续状态更新。
            6. 任务完成，B返回结果给A。
        - **生态：** 发布时有50家企业参与，30%为AI/平台服务，34%为软件SaaS/企业平台。
        - **ANP观点：** A2A主要解决企业内部智能体连接和协作问题，不适合互联网场景（隐私泄露：任务描述中包含隐私信息）。
    - **ANP (Agent Network Protocol) 是什么？**
        - **目标与愿景：** 成为智能体互联网时代的HTTP，定义智能体连接方式，构建开放、安全、高效的数十亿智能体协作网络。
        - **定位：** 全球首个面向智能体设计的开源通信协议，在智能体通信领域处于领先地位，尤其在智能体与互联网交互协作方面。
        - **核心目标：** 让任意两个智能体互联互通，构建AI原生数据网络。
        - **分层架构：** 分层分模块设计，核心三模块：
            1. **智能体身份 (Agent Identity):**
                - **解决问题：** 连接时“我是谁，你是谁”的问题。
                - **方案：** 基于W3C DID（去中心化身份ID）设计，具有互操作性的分布式身份认证方案。
                - **类比：** 类似于Email，拥有一个DID即可与全网智能体连接，无需注册账号。
                - **技术细节：** 基于Web技术，与云计算和域名绑定，不使用区块链。DID基于公私钥加密体系，私钥签名，公钥验证。公钥存储在域名下的DID文档中，B通过HTTP请求拉取A的公钥进行验证。
                - **扩展性：** 可服务数十亿用户。
            2. **智能体描述 (Agent Description):**
                - **解决问题：** 智能体如何公开自身能力和信息。
                - **方案：** 使用Web相关技术，如JSON-LD (Linked Data) 和Schema.org。
                - **原理：** 智能体可构建一个JSON格式的“首页”文档，描述开发者、能力、公开信息。通过URL将酒店房型、地址、图片、视频、接口等信息连接成一个AI友好的数据网络。
            3. **智能体发现 (Agent Discovery):**
                - **解决问题：** 其他智能体如何在互联网上找到我。
                - **方案：** 使用RFC 8615技术，类似于Web网站发现。
                - **两种机制：**
                    - **主动式 (搜索引擎):** 搜索引擎通过域名服务器找到域名，拼接`/well-known/agent.json`路径（基于RFC 8615扩展），爬取该域名下所有公开智能体及其信息，建立索引。
                    - **被动式 (主动注册):** 智能体主动注册到搜索服务。
            4. **元协议 (Meta Protocol):**
                - **目标：** 智能体可先用自然语言协商通信协议，然后各自生成协议代码进行交互。
                - **愿景：** 未来互联网协议很多将是智能体之间的共识协议。
        - **ANP交互流程 (以酒店预订为例):**
            1. **发现：** 智能体A通过搜索服务找到智能体B（酒店智能体）的描述文档URL。
            2. **认证与拉取：** 智能体A用私钥签名HTTP请求，携带DID信息。智能体B收到请求后，通过A的DID访问其DID文档获取公钥，验证签名。验证通过后，B将酒店描述、房型、价格等信息发送给A。
            3. **决策与预订：** 智能体A在本地分析信息，决策预订。如需预订，直接调用酒店智能体的API下单。
            4. **售后：** 可通过自然语言接口与酒店AI客服交互。
        - **基于ANP的智能体点对点交易流程：**
            - **核心：** 用户智能体直接与酒店智能体预订。
            - **必选服务：**
                - **支付智能体：** 托管资金，解决售后问题。
                - **仲裁智能体：** 解决交易纠纷，判断凭证信息。
            - **可选服务：**
                - **可信/公正服务：** 查询酒店信用分、评价分。
            - **凭证：** 使用W3C可验证凭证规范，保证交易链路凭证不可篡改。
        - **权限细分控制与隐私保护：**
            - **多DID隐私保护策略：**
                - **主DID：** 维持社交关系，仅给社交好友。
                - **子DID：** 购物、外卖、订票等场景使用不同子DID。全网无法跟踪主DID，无法形成完整用户画像，保护隐私。
                - **子DID周期性更新：** 即使子DID被截获，周期更新后也无法持续跟踪。
        - **重要权限的人类控制：**
            - **默认授权：** 智能体可自动爬取信息（如房间价格），无需人类授权，提高效率。
            - **人工授权：** 涉及付款等重要操作时，协议设计特殊字段，需人类生物识别（人脸、指纹）签名授权，私钥保存在个人助手无法访问的地方，确保资金安全。
        - **ANP能做什么 (酒店预订示例):**
            - 个人助手直接查询房间、预订酒店。

### [01:08:45 - 01:31:13] MCP、A2A、ANP协议对比

- **核心论点：** ANP在智能体身份、协议架构和设计理念上与MCP和A2A存在显著差异，更适合构建开放、去中心化的智能体互联网。
- **详细展开：**
    - **MCP与ANP的区别：**
        - **类比：**
            - MCP是模型的Type C接口（构建智能体）。
            - ANP是智能体的Email（智能体之间协作）。
        - **核心差异：**
            - **解决问题：** MCP解决构建智能体问题，ANP解决智能体之间协作问题。
            - **身份：**
                - MCP：基于OAuth 2.1的中心化身份方案，需在每个智能体系统注册账号。
                - ANP：基于W3C DID的去中心化身份，拥有自己的ID即可与全网智能体连接，无需注册。
                - **ANP观点：** 中心化身份方案在数十亿智能体互联互通时存在瓶颈，MCP和A2A未解决智能体身份核心问题。
            - **协议架构：**
                - MCP：典型CS架构，客户端主动连接服务器，服务器无法主动触达客户端（单向）。
                - ANP：原生P2P架构，任意智能体都可发起对对方的连接（双向）。
                - **ANP观点：** CS架构适合连接工具和资源，但不适合智能体之间协作。
            - **信息组织：**
                - MCP：JSON-RPC远程调用，获取资源/工具列表。
                - ANP：Linked Data技术，用URL将公开信息和接口连接成AI可访问的数据网络，复用Web基础设施（索引、DNS、CDN）。
                - **ANP观点：** ANP在处理智能体公开信息上更高效，与Web技术体系一脉相承。
            - **设计理念：**
                - MCP：以模型为中心 (Model-Centric)，整个互联网都是模型的上下文工具。
                - ANP：以智能体为中心 (Agent-Centric)，每个智能体地位同等，组成高效扁平的协作网络。
    - **A2A与ANP的差异：**
        - **身份：**
            - A2A：使用“代外”身份方案（协议之外解决身份问题），如通过三方身份系统获取令牌。适合企业内部（美国SaaS生态有完备身份授权系统）。
            - ANP：协议原生支持DID身份，协议内携带身份信息。也支持代外方案。
            - **ANP观点：** A2A未解决互联网上智能体通信的核心身份问题。
            - **类比：** A2A用介绍信，ANP自带身份证。
        - **核心概念：**
            - A2A：概念多且复杂 (Task, Message, Artifact, Part, Task State等7个状态)。擅长解决复杂任务交互，但跨域交互复杂度高。基于任务的交互可能泄露隐私。
            - ANP：核心概念只有两个 (Information, Interface)。Interface分自然语言和结构化两种。Information通过Linked Data组成网络。
            - **ANP观点：** A2A不适合在互联网这种不可信环境中使用，更适合企业内部可信环境。
    - **A2A vs ANP vs MCP 全局差异对比 (10个维度总结):**
        - **解决问题：**
            - A2A：企业内部智能体之间的复杂协作。
            - ANP：智能体在互联网上的连接与协作。
            - MCP：模型与工具、资源的连接。
        - **设计原则：**
            - A2A：复杂、复用现有协议；强调Task。
            - ANP：简单、复用现有协议；强调身份；注重隐私。
            - MCP：连接工具与资源；耦合大。
        - **协议架构：**
            - A2A：P2P。
            - ANP：P2P。
            - MCP：C/S。
        - **传输层：**
            - A2A：HTTP。
            - ANP：HTTP。
            - MCP：HTTP, stdio。
        - **核心概念：**
            - A2A：Task, Artifact, Message, Part。
            - ANP：Information, Interface。
            - MCP：Task, Resources, Sampling, Root, Prompts。
        - **智能体身份：**
            - A2A：OpenAPI生态的认证方式 (HTTP Basic, API Key, Cookie, OAuth 2.0, OpenID Connect)。
            - ANP：W3C DID为主。
            - MCP：OAuth。
        - **智能体描述：**
            - A2A：JSON (Agent Card)。
            - ANP：JSON-LD, schema.org。
            - MCP：未明确。
        - **智能体发现：**
            - A2A：RFC 8615, agent.json。
            - ANP：RFC 8615, agent-descriptions。
            - MCP：未明确。
        - **智能体通信模型：**
            - A2A：JSON-RPC。
            - ANP：Linked Data (语义网)。
            - MCP：JSON-RPC。
        - **开源许可：**
            - A2A：Apache 2.0。
            - ANP：MIT (计划变更为Apache 2.0)。
            - MCP：MIT。
        - **透明程度：**
            - A2A：灰盒。
            - ANP：黑盒。
            - MCP：白盒。
        - **适用场景：**
            - A2A：企业内部协作，也可用于智能体联邦，可用于企业内部及联邦。
            - ANP：智能体在互联网上的连接与协作。
            - MCP：模型工具/资源调用。
        - **对行业发展：**
            - A2A：短期内影响MCP，ANP将以行业标准为目标推动发展，已成为模型连接工具与资源的事实标准，让互联网平台在云端。

        - 左侧：MCP以模型为中心 (Model-Centric)，整个互联网都是模型的上下文和工具。
        - 右侧：ANP以智能体为中心 (Agent-Centric)，每个智能体具有同等的地位，组成一个高效、扁平的智能体协作网络。

### [01:31:13 - 01:43:49] 智能体互联网未来展望

- **核心论点：** 智能体互联网将构建AI原生数据网络，实现AI原生连接与协作，深刻改变软件开发、运行和连接方式，并促进消费互联网与产业互联网的深度融合，重构平台产业格局。
- **详细展开：**
    - **未来：AI原生数据网络**
        - **概念：** 现在的网络为人设计（HTML页面），AI访问低效。未来将出现方便AI访问的网络。
        - **架构：**
            - **个人智能体 (Personal Agent):** 每个人通过个人智能体上网，可能不止一个，且可互联。
            - **搜索智能体 (Search Agent):** 个人助手通过搜索智能体连接服务智能体。
            - **服务智能体 (Service Agent):** 对外提供公开服务，可连接上游供应链（产业网）。
            - **智能体数据网络 (Agent Data Network):** 智能体公开信息通过URL连接成AI原生数据网络。
        - **特点：**
            1. **专为AI设计：** 非为人设计。
            2. **半结构化数据：** JSON为主，内含自然语言供AI处理。
            3. **自描述能力：** 智能体能自行判断下一步应拉取哪些信息。
        - **基础设施：** 复用现有Web基础设施（DNS, CDN, PKI, CA证书, SSL/TLS, HTTP）。
    - **未来：AI原生连接与协作**
        - **模型驱动：** 协议无需手写代码，只需向模型提供提示词（如何使用ANP协议）和HTTP收发接口。
        - **案例：** 模型自动组装HTTP请求，首次忘记城市信息，收到500错误后，模型自我修复，再次构造请求带上城市信息，成功获取响应。
        - **影响：**
            1. **改变软件开发方式：** AI提升开发效率（如Copilot）。
            2. **改变软件运行方式：** 业务逻辑内化到模型，减少代码量。
            3. **改变软件连接方式：** 模型可即时修复错误，并将错误作为记忆，下次不再犯。
            4. **对职业生涯的影响：** 架构师、大厂工程师的职业生涯可能受影响，需提前考虑转型。
    - **未来：消费互联网与产业互联网融合**
        - **现状：** 消费互联网和产业互联网通过平台连接，连接带宽有限。
        - **未来：** 个人智能体直接连接产业互联网智能体，带宽大幅增加，实现深度融合。
        - **商业机会：** 融合将诞生大量商业机会。
        - **平台演变：** 互联网平台将演变为连接的工具（如搜索引擎），而非封闭平台（如微信）。
    - **平台产业格局演变：**
        - **之前：** 垂直整合格局（超级APP内部集成支付、物流、搜索等，消费者通过平台连接供给侧）。
        - **未来：** 智能体驱动平台产业水平解耦（消费者智能体直接通过支付工具、物流工具、搜索工具连接供给侧智能体）。
        - **引用：** 侯红文存（侯红老师，北京大学国家发展研究院助理教授，ANP生态战略顾问）。
