- **适用受众/场景：** 适用于希望了解和开发基于大模型（LLM）的智能体及多智能体协作系统的开发者、AI研究员、以及对自然语言编程和自动化软件开发感兴趣的技术人员。主要解决如何通过结构化的协作流程，让AI智能体高效完成复杂任务的问题。

## 2. 核心知识字典（Glossary）

-   **MetaGPT**: [00:12:37] 一个多智能体框架，它通过让不同的大模型驱动的智能体（Agent）进行协作来解决复杂问题，尤其擅长模拟软件公司进行代码编写，实现自然语言编程。
-   **Agent (智能体)**: [00:23:07] 在MetaGPT中，智能体被抽象为“角色”（Role）。它需要一个大模型（LLM）作为思考中枢，并配备观察环境、思考、行动和记忆的能力，以完成复杂任务。
-   **SOP (Standard Operating Procedure - 标准化操作流程)**: [00:14:49] MetaGPT引入的核心贡献，它将人类在软件开发等复杂任务中行之有效的工作流程（如产品经理分析需求、架构师设计、工程师编码、测试工程师测试等）融入到智能体协作中，使其能够更高效地完成复杂任务。
-   **Role (角色)**: [00:36:41] MetaGPT中对智能体的一种抽象，代表一个具有特定职责和能力的实体。一个Role可以被赋予一个或多个Action，并定义其观察（watch）、思考（think）和行动（act）的逻辑。
-   **Action (动作)**: [00:34:11] 在MetaGPT中，Action是智能体执行的具体操作，通常是对大模型（LLM）调用的一种封装，用于完成特定的子任务，如编写代码、运行代码、编写测试用例或进行评审。

### [00:32:40 - 00:43:15] 智能体入门：定义单个智能体

-   **核心论点：** 在MetaGPT中，定义一个智能体（Role）最简单的方式是定义其可执行的动作（Action），然后将这些动作赋予角色。一个Action是对LLM调用的封装，而Role则将多个Action串联起来，形成一个工作流程。
-   **详细展开：**
    *   **最简单的Agent**: 只需要定义好其“行动”（Action）即可，思考部分可以简化。
    *   **定义Action**:
        *   从 `metagpt.action` 导入 `Action` 类。
        *   创建一个继承 `Action` 的类，例如 `SimpleWriteCode`。
        *   重写 `_run` 方法，该方法内部通过 `self.llm.ask` 调用大模型API，并传入一个Prompt模板。
        *   `SimpleWriteCode` 的Prompt是“编写一个能计算列表乘积的Python函数，并提供运行代码”。
    *   **定义Role (角色)**:
        *   从 `metagpt.roles` 导入 `Role` 类。
        *   创建一个继承 `Role` 的类，例如 `SimpleCoder`。
        *   在 `__init__` 方法中，设置角色名称（`name`）、描述（`profile`）和目标（`goal`）。
        *   通过 `self.set_actions` 方法将 `SimpleWriteCode` 动作赋予 `SimpleCoder` 角色。
        *   重写 `_act` 方法，在该方法中调用 `self.rc.action` 来执行已定义的动作。
    *   **具有多个动作的智能体**:
        *   定义第二个Action，例如 `SimpleRunCode`，用于执行代码。
        *   创建一个新的Role，例如 `RunnableCoder`，它将 `SimpleWriteCode` 和 `SimpleRunCode` 两个动作都赋予自身。
        *   通过 `self.set_react_mode("by_order")` 指定动作按顺序执行，即先写代码，再运行代码。
    *   **运行演示**:
        *   运行 `RunnableCoder` 角色，输入需求“编写一个计算列表乘积的函数”。
        *   第一步：`SimpleWriteCode` 生成Python函数代码。
        *   第二步：`SimpleRunCode` 执行生成的代码，并输出结果（例如 `[1,2,3,4]` 的乘积是24，`[7,2,9]` 的乘积是504）。
    *   **智能体内部运行流程回顾**:
        *   **Observe (观察)**: 接收用户需求或环境消息。
        *   **Think (思考)**: 决定要使用哪个Action。
        *   **Act (行动)**: 调用选定的Action执行任务。
        *   **Publish (发布)**: 将行动结果发布到环境中。
-   **重要金句/原话：** “一个agent他需要有个LLM作为他一个思考的中书，但除此之外我们要给他配备观察环境的能力、思考的能力以及行动的这样一些以及记忆的这样些能力，他才能够作为一个可以就是类比我们人类的这样一个智能体去做一些复杂的事情。”

### [00:44:16 - 01:02:25] 多智能体开发：协作与消息机制

-   **核心论点：** 多智能体通过环境进行消息交互，每个智能体根据其“关注”（watch）的消息进行思考和行动，并将结果“发布”（publish）到环境中，形成一个SOP驱动的协作流程。
-   **详细展开：**
    *   **多智能体交互机制**:
        *   每个Agent内部仍遵循 Observe -> Think -> Act 的循环。
        *   新增 `publish_message` 动作，将结果发布到环境。
        *   `observe` 动作从环境中获取消息。
    *   **“软件公司”案例**: 模拟一个更复杂的软件开发流程，包括代码编写、测试用例编写和测试用例评审。
        *   **需求**: 给定一个资源描述，智能体需要完成代码编写、执行代码、生成测试用例，并对测试用例进行评审。
        *   **角色分工**:
            *   `SimpleCoder`: 负责编写代码。
            *   `SimpleTester`: 负责编写测试用例。
            *   `SimpleReviewer`: 负责评审测试用例。
        *   **SOP流程**:
            1.  User发布需求 (`user requirement`) 到环境。
            2.  `SimpleCoder` 观察到需求，执行 `WriteCode` 动作，将生成的代码发布到环境。
            3.  `SimpleTester` 观察到 `SimpleCoder` 发布的代码，执行 `WriteTest` 动作，生成测试用例并发布到环境。
            4.  `SimpleReviewer` 观察到 `SimpleTester` 发布的测试用例，执行 `WriteReview` 动作，给出评审意见并发布到环境。
    *   **代码实现**:
        *   定义 `WriteCode`, `WriteTest`, `WriteReview` 三个Action。
        *   定义 `SimpleCoder`, `SimpleTester`, `SimpleReviewer` 三个Role。
        *   **关键点 `watch`**: 每个Role通过 `self.set_rc_actions(..., watch=[Message])` 指定其关注的消息类型。
            *   `SimpleCoder` 关注 `BossRequirement`。

### [01:02:26 - 01:18:35] 思考与行动机制：三种反应模式

-   **核心论点：** MetaGPT提供了三种智能体反应模式：React（默认，动态决策）、Act by Order（顺序执行，静态SOP）和Plan and Act（规划后行动，未来功能），以适应不同复杂度和确定性的任务场景。
-   **详细展开：**
    *   **三种反应模式**:
        1.  **标准React模式 (默认)**:
            *   参考React论文实现。
            *   适用于Agent需要根据环境或观察到的消息动态决定下一步行动的场景。
            *   内部通过LLM进行动作选择。
            *   流程：Agent观察到消息 -> `think` 阶段确定要使用的Action -> `act` 阶段调用选定的Action -> 输出结果。如果输出改变环境，则进入下一个循环，重新思考和选择动作。
            *   特点：动态、灵活，但可能消耗更多Token且稳定性略低。
            *   可通过 `max_iters` 参数控制 `think-act` 循环次数。
        2.  **按顺序执行 (Act by Order) 模式**:
            *   适用于SOP（标准化操作流程）已确定、静态的场景。
            *   Agent按照预设的Action顺序依次执行。
            *   优点：更简洁、稳定，避免了每次都用LLM思考动作选择的开销和不稳定性，节省Token和时间。
            *   示例：`RunnableCoder` 先 `SimpleWriteCode`，再 `SimpleRunCode`。
            *   在 `__init__` 中通过 `self.set_react_mode("by_order")` 声明。
        3.  **规划后行动 (Plan and Act) 模式 (未来功能)**:
            *   模拟人类先规划后行动的模式，而非“走一步看一步”。
            *   适用于问题解决流程不确定，但需要长期规划能力的场景。
            *   流程：Agent首先生成一个计划（plan），然后根据计划逐步调用Action执行。
            *   优点：赋予Role更长期的规划能力，提高复杂任务的完成效果。
            *   目前代码尚未完全开源，但将在近期迭代中提供。
