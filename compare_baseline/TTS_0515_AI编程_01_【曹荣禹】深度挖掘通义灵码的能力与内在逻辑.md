## 2. 核心知识字典（Glossary）
- **通义灵码 (TONGYI Lingma):** 阿里巴巴统一实验室开发的一款智能编码助手，旨在通过AI技术提升开发者的编程效率，提供代码生成、补全、智能问答、AI程序员等功能。
- **代码生成补全 (Code Generation & Completion):** 根据当前代码上下文，预测并推荐接下来可能编写的代码片段，包括行级、块级、函数级甚至类级的代码，是AI编码助手最核心的功能之一。
- **AI程序员 (AI Programmer):** 通义灵码推出的一个产品，能够根据用户提出的自然语言需求（如功能开发、缺陷修复），自主规划、分析项目、定位代码、生成修改建议并以标准Diff形式提交代码，实现端到端的项目修改流程。
- **RAG (Retrieval-Augmented Generation):** 检索增强生成技术，在代码生成补全场景中，指通过检索项目内相似代码片段或相关知识，作为上下文输入给大模型，以提高生成代码的准确性和相关性，避免幻觉。
- **Codev-Bench:** 阿里巴巴通义灵码团队提出的一个面向用户角度的代码补全评测集，旨在更真实地评估代码大模型在多样化编程场景下的补全能力，通过单元测试的执行通过率作为评估指标。
- **SWE-bench:** 一个在学术界广泛使用的基准测试集，用于评估AI在软件工程任务（如修复GitHub上的真实缺陷或实现功能需求）中的能力，其评估标准是能否通过项目原有的单元测试。

### [00:18:27 - 00:31:27] 编码助手中代码生成补全的探索
- **核心论点：** 代码生成补全作为AI编码助手的核心刚需场景，面临多语言、多框架、复杂上下文等挑战。通义灵码通过训练代码专属大模型，并结合RAG技术实现项目级代码理解，以提供更精准、无幻觉的补全。
- **详细展开：**
    - **代码生成补全的刚需性：**
        - 示例：在编写代码时，编码助手能根据上下文自动推荐行级或块级代码，提升效率。
    - **代码生成补全的困难与挑战：**
        1.  **多语言与多框架：**
            - 需要支持Java, Python, Go等多种编程语言，以及各种框架和规范。
            - 存在长尾语言问题，且每种语言/框架的知识密度高。
        2.  **复杂上下文：**
            - 不仅要考虑当前文件，还要理解项目中其他文件定义的类和函数，避免模型生成幻觉。
            - 需学习庞大的二方库和三方库（不同版本有不同特性），对模型学习造成巨大困难。
    - **业界解决方案：代码专属大模型**
        - **核心思路：** 基于通用语言模型，通过海量真实代码数据训练出代码专属大模型（如Llama -> Code Llama, Qwen -> Code Qwen）。
        - **训练要素：**
            - **海量真实代码数据：** 从GitHub、Gitee等平台获取真实仓库数据，学习各种语言、框架、二方/三方包。
            - **Fill-in-the-Middle (FIM) 训练模式：** 与通用语言模型（next token prediction）不同，代码补全需要同时考虑光标位置的上文和下文来补全中间代码。
            - **整库感知的数据组织方式：** 补全时需参考整个项目库中定义的类和函数，避免生成幻觉。
    - **超视距项目级别代码理解（6点可感知）：**
        - **本文件：**
            - 丰富的上下文场景：用户需求多样（for循环、if判断、函数、类），生成粒度多样（行、块、函数）。
            - 长文件处理：对文件中定义的函数进行归纳总结，帮助模型理解。
            - 用户行为建模：预测用户意图，如长函数只生成签名，避免长时间等待和不必要的干扰。
        - **跨文件：**
            - 解析跨文件的类/函数定义：理解项目中其他文件定义的类、函数签名，并进行简单概括。
            - 构造跨文件调用学习样例：训练模型生成跨文件调用的能力。
            - 示例：在`DWCReportImportServiceImplement.java`中补全`DWCReport`对象的方法时，能正确识别`DWCReport.java`中定义的字段和函数，避免幻觉。
        - **本仓库：**
            - 利用RAG参考项目内相似代码：对仓库文件进行分块、向量化，根据用户当前补全上下文构造检索请求，从向量库中检索相似代码片段，作为提示信息辅助模型生成。
        - **三方包：**
            - 解析三方包依赖：解析当前仓库的框架信息、依赖的三方包及其版本，构造数据让模型学习，提高对三方包的理解。
        - **通义灵码已具备的4点可感知能力：** 补全位置可感知、项目中调用文件可感知、相似代码可感知、三方包依赖可感知。
        - **未来展望的2点可感知能力：** 数据结构可感知（理解对象所属类、类内函数/字段）、执行环境可感知（理解项目安装环境，生成更精准代码）。
